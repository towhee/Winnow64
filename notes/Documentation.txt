INITIALIZATION

    Persistent settings are saved between sessions using QSettings. Persistent settings
    form two categories at runtime: action settings and preference settings. Action
    settings are boolean and maintained by the action item's setChecked value.
    Preferences can be anything and are maintained as public variables, mostly in MW
    (MainWindow) and managed in the PrefDlg class.

    Not all settings can be read immediately, as the classes that use them may not yet
    have been created. Examples are IconView parameters and the recent folders menu
    list, since IconView and the menus have not been created yet. Therefore some
    settings are assigned up front and the rest are assigned as needed.

------------------------------------------------------------------------------------------
Startup Sequence
------------------------------------------------------------------------------------------

    • Load QSettings
    • Set default values if no settings available (first time run)
    • Set preference settings including:
        • General (previous folder etc.)
        • Slideshow
        • Cache
        • Full screen docks visible
        • List of external apps
        • Bookmarks
        • Recent folders
        • Ingest history folders
        • Workspaces
    • Create actions and set checked based on persistent values from QSettings
    • Create bookmarks with persistent values from QSettings
    • Update external apps with persistent values from QSettings
    • Load shortcuts (based on being able to edit shortcuts)
    • Execute updateState function to implement all persistent state settings

    • Select a folder
        • Load datamodel with QDir info on each image file
        • Add the rest of the metadata to datamodel (incl. thumbs as icons)
        • Update the image cache

Dock Window State (versioning)
------------------------------------------------------------------------------------------

    The QMainWindow dock layout is persisted with saveState()/restoreState() under QSettings
    "WindowState" (MW::settings writes it; MW::showEvent restores it). Both calls pass
    MW::winnowStateVersion.

    RULE: BUMP winnowStateVersion whenever the SET of docks changes (a dock added or removed).
    A WindowState saved by an older build does not contain the new dock, and restoring it
    leaves that dock's group orphaned as empty "zombie" tab bars; the dock layout then never
    converges and the tabbed docks flicker continuously. The version tag makes restoreState()
    reject the stale state (returns false, changes nothing) so the layout built in
    MW::createDocks/initialize() stands; the state is re-saved at the current version on exit,
    so it self-heals after one launch (cost: a one-time reset of dock sizes/positions).

    Because that fallback layout is what shows after a reject, keep it sane: do NOT tabify a
    dock in one area with a dock in another area (e.g. the left-area metadataDock with the
    right-area embelDock) -- a cross-area tabify drags both into one area and crams every dock
    into a single tab group, where the raised tab oscillates against the squeezed central
    widget. embel + develop tab together on the RIGHT (see initialize() tabifyDockWidget calls).

    developDock (added with the raw editor) was the first dock to trip this: the pre-existing
    saved state predated it AND, once saved during a flicker, the corrupt state perpetuated
    itself across launches until the version bump discarded it.

FOLDER LOADING PIPELINE

    Use G::isFlowLogger or G::isLogger to trace program progress.

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    Loading a folder runs in two phases that do NOT overlap with each other.
    Phase 1 populates the DataModel with one row per eligible file and runs on
    the GUI thread. Phase 2 reads metadata and icons (MetaRead, on worker
    threads) and caches full-size images (ImageCache, on worker threads); these
    two activities within Phase 2 DO overlap.

    Phase 1 - Build the DataModel (GUI thread)

    • A folder or bookmark click calls MW::folderSelectionChange. The mouse
      modifier decides whether the folder replaces the current selection
      (resetDataModel = true) or is added to it.

    • When resetDataModel is true the prior load is torn down first: MW::stop
      halts the worker threads, MW::reset clears state, and
      DataModel::clearDataModel empties the model. clearDataModel calls
      QCoreApplication::removePostedEvents(this, QEvent::MetaCall) BEFORE
      clear(). QStandardItemModel coalesces item edits into a posted
      _q_emitItemChanged meta-call; if that (or a stale worker setData/setIcon
      meta-call) fired after clear() freed the items it would dereference freed
      pointers and crash. DataModel::newInstance then bumps dm->instance so any
      surviving stale worker signals are ignored.

    • Subfolders are counted (Utilities::subFolderTree) and the folder plus,
      in the recursive case, every subfolder are queued by
      DataModel::enqueueFolderSelection / enqueueOp. macOS .photoslibrary
      bundles are filtered out (they hold huge numbers of derivative files).

    • DataModel::scheduleProcessing starts DataModel::processNextBatch, which
      drains the folder queue in time-sliced batches. It yields to the event
      loop every ~30 ms (kTimeBudgetMs), with a secondary cap of 64 folders per
      tick, rescheduling itself with a zero-delay timer. This time-slicing is
      what keeps the GUI responsive while a large recursive tree loads.

    • DataModel::addFolder handles one directory: dir.entryInfoList() lists
      the eligible files, they are sorted (DataModel::lessThan, or
      lessThanCombineRawJpg when raw+jpg combining is on), and each file is
      appended as a row (setRowCount + DataModel::addFileDataForRow, which fills
      the OS file columns and performs raw+jpg pairing).

      Cost note: the sort/filter proxy (dm->sf) inserts each new row into sorted
      position at O(N), so the whole load is O(N^2). Time-slicing bounds each
      GUI freeze to roughly one folder's insert time but does not change that
      total; a single very large folder loaded late can still block briefly.

    • When the queue empties DataModel emits folderChange, handled by
      MW::folderChanged. It re-enables the panels, sets the current index, and
      either finishes immediately (DataModel::isAllMetadataAttempted, e.g. a
      pure removal with no new rows) or starts Phase 2.

    Phase 2 - Read metadata/icons and cache images (worker threads)

    • MW::folderChanged starts MetaRead with a queued
      QMetaObject::invokeMethod(metaRead, "initialize") followed by
      MetaRead::setStartRow(startRow). MetaRead lives on metaReadThread.

    • MetaRead::dispatchReaders / dispatch assign rows to Reader workers,
      walking outward from startRow (the current image) so the user's position
      is served first. Each Reader reads metadata (Reader::readMetadata ->
      DataModel::addMetadataForItem) and a G::maxIconSize (256 px) icon
      (Reader::readIcon -> DataModel::setIcon1, stored in Qt::DecorationRole),
      delivered to the GUI thread over queued connections. Backpressure
      (dm->queuedReaderEvents, cap 4 x readerCount) defers dispatch when the GUI
      thread falls behind draining those events.

    • Once the start row itself has been read, MetaRead emits select, which
      routes Selection::setCurrentIndex -> MW::updateChange ->
      MW::fileSelectionChange. That loads the current image and starts
      ImageCache, so ImageCache runs concurrently with the rest of MetaRead.

    • ImageCache caches full-size images ahead of and behind the current
      position: ImageCache::setCurrentPosition -> updateTargets ->
      launchDecoders -> decodeNextImage -> ImageDecoder (decode / rotate /
      colorManage) -> ImageCache::fillCache -> cacheImage -> nextToCache.

    • MetaRead::done signals MW::folderChangeCompleted, which builds the
      filters, re-enables the sort/filter menus, applies a single sort pass, and
      resizes the table columns.

    Instances and aborting

    • Every new primary folder increments dm->instance
      (DataModel::newInstance). Worker signals carry the instance they were
      issued under; DataModel slots (setIcon1, addMetadataForItem, setValSf,
      ...) drop any call whose instance no longer matches, so lagging signals
      from an abandoned load are harmless.

    • G::stop and dm->abort are the cooperative stop flags. MW::stop sets
      them, signals the workers to abort, and waits (event-driven, 3 s cap)
      for them to acknowledge before resetting.

------------------------------------------------------------------------------------------
Concurrent Flow - New Folder Selected
------------------------------------------------------------------------------------------

    • Selecting a new folder in the Folders or Bookmarks panel calls
      MW::folderSelectionChange.

    • If the folder replaces the current selection, the prior load is torn
      down: all worker threads are stopped, the DataModel is cleared and all
      parameters are reset (MW::stop, MW::reset, DataModel::clearDataModel).

    • The DataModel is loaded from the folder queue by
      DataModel::processNextBatch, which calls DataModel::addFolder and
      iterates DataModel::addFileDataForRow to populate basic file info per
      row. processNextBatch is time-sliced (yields every ~30 ms) so the GUI
      stays responsive. MW::folderChanged then sets the current index.

    • MetaRead::setStartRow / dispatchReaders / dispatch send Reader workers
      outward from the start row, loading metadata
      (-> DataModel::addMetadataForItem) and 256 px icons
      (-> DataModel::setIcon1).

    • When the start row has been read, MetaRead emits select, which (via
      MW::updateChange -> MW::fileSelectionChange) starts ImageCache. From
      there ImageCache walks setCurrentPosition -> updateTargets ->
      launchDecoders -> fillCache, caching concurrently with MetaRead.

------------------------------------------------------------------------------------------
Concurrent Flow - New Image Selected
------------------------------------------------------------------------------------------

    A new image is selected, which triggers fileSelectionChange:

    • If starting the program, select the first image in the folder.
    • Record the current datamodel row and its file path.
    • Update the thumbView and gridView delegates.
    • Synchronize the thumb, grid and table views. If in loupe mode, load
      the current image.
    • Update window title, statusbar, info panel and classification badges.
    • Update the metadata and image caching.
    • If the metadata has not been cached yet for the selected image
      (usually the first in a new folder), load the thumbnail.
    • Update the cursor position on the image caching progress bar.

    A new image is selected which triggers a scroll event:

    • IconView selection change may trigger a scroll signal, based on the
      current view (thumb, grid or table). MW::thumbHasScrolled /
      gridHasScrolled / tableHasScrolled are signalled.
    • Update the icon range (firstVisible / lastVisible).

------------------------------------------------------------------------------------------
Icon Chunk Size
------------------------------------------------------------------------------------------

    Icons or thumbnails, 256 px on the long side, are stored in the DataModel
    in column 0 under Qt::DecorationRole (the QPixmap); the icon's QRect is
    kept on the same cell under G::IconRectRole. The number of icons to cache is
    dm->iconChunkSize, set per folder in DataModel::resolveIconChunkSize: in brute
    force (G::useJitIconCache == false) it equals the row count (all icons); in JIT
    it is the memory budget (iconBudgetCount), clamped to the row count and to the
    3x-visible floor (iconChunkFloor). See the "Icon Caching" section below for the
    full brute-force vs JIT model (Layers 1-3). As the user moves through the images,
    either by selection or scrolling, the icon range is recalculated based on the
    midVisibleIcon, and iconChunkSize is held at the floor by MW::updateIconRange.

    When the user progresses through the images, the current image is centred
    in the visible views (thumbView, gridView and tableView) using the
    midVisibleCell.

------------------------------------------------------------------------------------------
Thumbnail (Icon) Decoding
------------------------------------------------------------------------------------------

    Thumb::loadThumb (Image/thumb.cpp), called by Reader::readIcon, decodes the
    icon for column 0. It prefers a small embedded preview over decoding the full
    image. The routing is:

        • video                          -> loadFromVideo
        • isEmbeddedThumb (offset+len)   -> loadFromJpgData
        • cr3 (if loadFromJpgData fails) -> loadFromImageIO   (CR3 HDR PQ, see below)
        • heic                           -> loadFromHeic
        • tif (G::useMyTiff)             -> loadFromTiff
        • else                           -> loadFromEntireFile (Qt, scaled)

    Embedded thumbnail format by file type (verified against external format
    documentation - ExifTool, the DNG spec, and lclevy/canon_cr3):

        • JPEG and the raw formats NEF, CR2, ARW, SR2, RAF, ORF, RW2: the embedded
          thumb is JPEG (TIFF tags 513/514 or makernote payloads).
        • CR3: standard CR3 stores JPEG in the THMB (thumbnail) and PRVW (preview)
          boxes. BUT a CR3 shot in HDR PQ mode encodes BOTH THMB and PRVW as HEVC
          (hvc1) - a headerless HEIF bitstream, not JPEG. See "CR3 HDR PQ" below.
        • DNG: the spec allows preview/thumbnail IFDs to be Uncompressed (RGB), JPEG
          or JPEG XL. Winnow's DNG parser scans for JPEG streams and selects the
          smallest JPEG, so m.offsetThumb/lengthThumb always point at a JPEG.
          The DNG parser deliberately does NOT scan the Photoshop IRB (tag 34377)
          for a thumbnail the way Tiff::parse does: DNG stores its previews in
          subIFDs (tag 330), which the parser already handles. An IRB thumb only
          exists if the DNG was round-tripped through Photoshop, so it would at most
          be a fallback when no subIFD JPEG preview is present (rare). ifdPhotoshopOffset
          is still read in dng.cpp so IRB::readThumb() can be added there as a
          !m.lengthThumb-gated fallback if ever needed.
        • TIFF: IRB (Photoshop) thumbs are JPEG (m.thumbFormat = Jpg); chained-IFD
          and subIFD thumbs are TIFF format (m.thumbFormat = Tif). The TIFF-format
          thumbs leave m.lengthThumb == 0, so isEmbeddedThumb is false and they are
          handled by loadFromTiff rather than loadFromJpgData.
        • HEIC / HIF (incl. Canon HEIF stills): handled natively by loadFromHeic
          (libheif on Windows, ImageIO on macOS).
        • The remaining supported types (bmp, png, gif, webp, jp2, tga, svg, etc.)
          have no embedded thumbnail; they are decoded whole by loadFromEntireFile.

    loadFromJpgData reads m.offsetThumb/m.lengthThumb and decodes with the fast
    "JPEG" hint, then falls back to Qt content-sniffing (loadFromData with no format
    hint) if that fails. The fallback makes the path tolerant of any Qt-decodable
    self-contained embedded thumbnail; undecodable buffers return false and fall
    through to the next loader (loadFromTiff / loadFromEntireFile).

    CR3 HDR PQ: the THMB payload is a bare HEVC bitstream (no HEIF file header), so
    neither the "JPEG" decode nor the content-sniff fallback can read it and
    loadFromJpgData fails. loadThumb handles this with a failure-driven fallback:
    when ext == "cr3" and the embedded path did not succeed (and is not retrying a
    locked file), it calls loadFromImageIO before the loadFromEntireFile catch-all.

        • macOS: loadFromImageIO -> macImageIOThumbnail (ImageIO natively supports
          CR3 and returns the embedded preview, or a downscaled decode of the raw).
          This is the supported, tested platform.
        • Windows: loadFromImageIO returns Fail (no safe path - libheif rejects CR3
          and Heic::decodeThumbnail does not guard read failures), so loadThumb
          falls through to loadFromEntireFile as before. A full Windows fix would
          need WIC plus the Canon raw codec.

    This is failure-driven by design: standard CR3 files decode in loadFromJpgData
    and never reach loadFromImageIO, so their behaviour is unchanged. It also does
    not depend on parser-side detection of the hvc1 codec, which keeps the CR3
    parser untouched.

Tracking Icon Parameters
------------------------------------------------------------------------------------------

    The number of visible icons (visibleCellCount) can change whenever:

        • the size of the view changes
        • the size of the icons changes

    When this happens:

        • IconView::updateVisible is called. A number of IconView operations
          use the parameters defined in IconView::updateVisible.
        • MW::updateIconRange updates the DataModel iconChunkSize based on
          which view has the most visible icons and the current iconChunkSize.

    The first, mid and last visible icons change whenever there is a scroll
    event. The scroll event can be triggered by the user scrolling or
    selecting another image. See also SCROLLING below.

        • MW::thumbHasScrolled, MW::gridHasScrolled or MW::tableHasScrolled is
          triggered.
        • MW::updateIconRange calls IconView::updateVisible.
        • Other visible views are scrolled to sync.

Combine Raw+JPG
------------------------------------------------------------------------------------------

    When a folder holds raw+jpg pairs (same base name), the "Combine Raw+JPG" toggle
    decides whether each pair is shown as a single combined row (raw is primary, the jpg
    folded into it) or as two separate rows. The state lives in the bool combineRawJpg,
    which is mirrored to G::combineRawJpg, dm->sf, fsTree and bookmarks (the folder and
    bookmark panels show image counts that depend on the setting).

    Entry points (both route to MW::setCombineRawJpg):
        • combineRawJpgAction — File menu item, shortcut Opt/Alt + J. Being checkable,
          the action flips its own checked state before triggered fires.
        • rawJpgStatusBtn — status-bar button. Its lambda calls
          combineRawJpgAction->toggle() (which does not re-emit triggered) then calls
          setCombineRawJpg directly.

    combineRawJpgAction->isChecked() is the source of truth; setCombineRawJpg reads it
    into combineRawJpg. Because the trigger has already flipped the action before
    setCombineRawJpg runs, any early return must restore the action with
    setChecked(combineRawJpg) to keep the action and flag in sync — otherwise they
    desync and a later click is spent re-aligning them instead of toggling.

    Three load states:
        • Folder loaded (dm->rowCount() && G::allMetadataAttempted): full apply — persist
          to QSettings, rewrite the Type column for jpg duplicates ("JPG" vs "JPG+<raw>"),
          dm->rebuildTypeFilter, filterChange (re-pairs/splits the proxy) and
          buildFilters->updateAllCounts (the proxy baseline changed without a
          Filters-tree change, so unfiltered counts must be recomputed too).
        • No folder loaded (dm->rowCount() == 0): toggling is allowed; setCombineRawJpg
          records the setting and refreshes the folder/bookmark counts, then returns
          before the datamodel/proxy rebuild.
        • Folder loading (rows exist but !G::allMetadataAttempted): toggling is blocked.
          rawJpgStatusBtn and combineRawJpgAction are disabled in MW::folderChanged at
          load start and re-enabled in MW::folderChangeCompleted; updateStatusBar also

Folder Tree Auto-Refresh
------------------------------------------------------------------------------------------

    The Folders panel (FSTree) updates itself when subfolders are added or removed on
    disk outside Winnow (e.g. in Finder/Explorer) — no manual Refresh is needed.

    Mechanism: a cross-platform QFileSystemWatcher (FSTree::folderWatcher) wired up in
    FSTree::setupFolderWatcher(). Qt maps this onto the native OS facility (FSEvents on
    macOS, ReadDirectoryChangesW on Windows, inotify on Linux), so one code path serves
    both platforms.

    Scope: QFileSystemWatcher only reports paths explicitly registered, so rather than
    watch the whole filesystem Winnow watches only the folders currently visible in the
    viewport. updateWatchedFolders() diffs the visible set against the watched set and
    applies the delta; scheduleWatchResync() re-runs it (debounced 150 ms) on scroll,
    expand, collapse, selection, resize and refresh.

    On a directoryChanged signal, onWatchedDirectoryChanged() debounces (300 ms) then
    calls FSFilter::refresh() — which uses invalidate() so the view re-queries the live
    FSModel::hasChildren() and the dropdown decoration appears/disappears — and re-syncs
    the watch set so a newly created subfolder is itself watched.

    Note: drive/volume mount/unmount is a separate path (Windows WM_DEVICECHANGE via
    WinNativeEventFilter); this section covers folder content changes only.

    Key files: File/fstree.cpp / File/fstree.h.

CACHING

    Metadata caching has been replaced by `MetaRead` (`Cache/metaread.cpp`) dispatching a
    pool of `Reader` workers (`Cache/reader.cpp`). The earlier `MetadataCache` /
    `metadataCacheThread` system (`Cache/mdcache.cpp`, `Cache/mdcacher.cpp`,
    `Cache/mdcachemgr.cpp`) is no longer compiled — those files are not listed in
    `winnow64.pro` and are retained on disk for reference only.

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    • When a new folder or a new file is selected, or when a scroll event occurs in
      IconView, MetaRead is started. MetaRead dispatches Reader workers to load
      metadata and icons into the DataModel.

    • As metadata accumulates, MetaRead signals ImageCache::addCacheItemImageMetadata
      so the image cache can begin loading full-size images. When MetaRead crosses
      imageCacheTriggerCount it signals the ImageCache to start.

    • If a new image is selected, MW::fileSelectionChange repositions MetaRead and the
      ImageCache around the new current row.

    • If the user scrolls in the thumbView or gridView, scroll events are sent to
      MW::thumbHasScrolled / MW::gridHasScrolled. A delay ensures that rapid scrolling
      coalesces — the next scroll event supersedes the prior, and MetaRead is only
      re-dispatched when there is a pause or the scrolling is slow. ImageCache is not
      restarted here because a new image has not been selected.

    See `Cache/metaread.cpp` and `Cache/reader.cpp` top-of-file comments for more detail.

------------------------------------------------------------------------------------------

------------------------------------------------------------------------------------------
Icon Caching
------------------------------------------------------------------------------------------

    • Each icon is a 256 px longside QIcon, so we want to balance how many we load when
      weighing performance vs memory footprint.

    • Icons are stored in the DataModel at dm->(row, 0).data(Qt::DecorationRole). The
      loaded status can be determined two ways:
        1. dm->itemFromIndex(dmIdx)->icon().isNull()      // used up to 2023
        2. dmIdx.data(Qt::DecorationRole).isNull()        // switched to in 2023

    • The number of icons to cache is governed by dm->iconChunkSize. How it is set is
      decided per folder in DataModel::resolveIconChunkSize (called from endLoad) and
      depends on the G::useJitIconCache flag.

    Brute force (G::useJitIconCache == false, the default)

        • dm->iconChunkSize is set to the DataModel row count, so every icon is cached
          regardless of folder size. Simplest behaviour; preferred for typical folders.
          (The G::memoryOverrunFlag bail in DataModel::setIcon1 remains the crash backstop.)

    Just-in-time, JIT (G::useJitIconCache == true)

        • A folder is cached fully only when its projected thumbnail footprint fits a memory
          budget; otherwise dm->iconChunkSize is clamped to a bounded sliding window and the
          existing machinery (setIconRange, MetaRead::needToRead, clearIconsOutsideChunkRange)
          loads and evicts icons as the user moves. JIT has three layers:

        Layer 1 - predictive budget (resolveIconChunkSize / iconBudgetCount)
            At folder load the per-icon footprint is estimated (worst case 256x256x4 bytes)
            and compared to a budget = free memory after a safety reserve and the image
            cache's own remaining claim (G::imageCacheHeadroomMB, published by
            ImageCache::memChk), times G::jitIconCacheMemFraction. Fits -> full; else ->
            bounded window. iconChunkSize is then clamped to the row count and to a hard
            floor (iconChunkFloor): max(256, 3 x visibleIcons). The floor OVERRIDES the
            budget — if the image cache has claimed most of memory and the budget would fall
            below 3x the visible icons, the floor wins and the thumbnails take the memory
            they need; the image cache self-throttles because its own ceiling is derived
            from G::availableMemoryMB, which drops as the thumbnails are held. The floor is
            re-applied by MW::updateIconRange whenever the visible count changes (e.g.
            switching to the denser Grid view). Note: sizing is purely budget + floor; there
            is no separate "visible x 3" target beyond this floor.

        Layer 2 - measured refinement (refineIconChunkSize)
            As real icons load, DataModel::setIcon1 accumulates their true size. After ~200
            icons the budget is recomputed with the measured average and the window may grow,
            or be promoted back to full, when thumbnails turn out smaller than the worst-case
            estimate. Grow-only and one-shot per folder.

        Layer 3 - defensive pressure valve (applyIconCachePressure)
            A 1 s GUI-thread timer watches memory pressure (DataModel::memoryPressureLevel,
            derived from G::availableMemoryMB). On warn it halves the window; on critical it
            clamps to the visible page and evicts. It never grows. A 60 s cooldown plus a
            2 GB available-memory high-water mark (hysteresis) gate when the latch releases,
            so the cache cannot thrash between shrink and re-grow. Releasing does not re-grow;
            Layer 1/2 re-decide on the next folder load. Brute force is not touched by the
            valve.

    • Independent of the above, MW::updateIconRange still grows dm->iconChunkSize to at least
      the number of visible icons when more thumbnails are visible (in thumbView, gridView or
      tableView) than the current chunk, e.g. after an app or thumbnail resize.

    • When dm->iconChunkSize changes or there is a scroll event, the metadata cache read
      process is rerun to update the icons.

------------------------------------------------------------------------------------------
Load Responsiveness (keeping the GUI usable during a large/recursive load)
------------------------------------------------------------------------------------------

    The cost of a folder load on the GUI thread is dominated not by decoding but by the
    flood of QStandardItemModel signals each row change emits, every one of which runs the
    proxy (SortFilter) plus all three views' (thumbView, gridView, tableView) handlers
    synchronously. On a ~9000-image recursive tree this was ~30 s of GUI-thread time,
    starving keystrokes/mouse for seconds. Three measures cut it to near zero:

    1. Batch per-row writes (always on). DataModel::setIcon1 and setValDm wrap their setData
       calls in a QSignalBlocker and emit ONE dataChanged, mirroring addMetadataForItem.
       Four/two signals per icon -> one.

    2. Notify only visible rows (G::useVisibleOnlyIconEmit, default true). setIcon1/setValDm
       emit their one dataChanged only when the row is within [firstVisibleIcon,
       lastVisibleIcon] (DataModel::iconRowVisible). Off-screen icons are stored without
       notification and paint correctly when scrolled to. ~9000 emits -> ~visible-count.

    3. Batch the folder insert (G::useBatchedFolderInsert, DEFAULT FALSE). DataModel::addFolder
       can add a folder's rows with a single setRowCount (one rowsInserted, one view relayout)
       and fill the cells with signals blocked + one dataChanged for the range, instead of
       one rowsInserted + ~20 dataChanged per file (the enumeration freeze).

       The Z-A reordering that previously made this unusable is now fixed: while a batched
       load runs, DataModel::scheduleProcessing turns the proxy's dynamic sort/filter OFF
       (sf->setDynamicSortFilter(false)), so the one wide dataChanged per folder no longer
       re-sorts the inserted block. Inserted rows map in source (lessThan / name) order;
       DataModel::restoreProxySortAfterLoad re-enables dynamic sort in one pass when the
       load finishes or aborts (every processNextBatch exit path), restoring the same final
       order the per-row path produces. Still DEFAULT FALSE pending A/B measurement on the
       recursive pictures tree (see G::isPerfProbe below); flip it to true to test.

    Items 1-2 are independent of brute force vs JIT — they help any mode. Set the flags
    false to restore the prior per-signal behavior.

      Phase 1 (build DataModel, GUI thread): baseline wall 2408 ms -> 542 ms (4.4x) after
      three fixes. Instrumentation overturned the prior assumption that the proxy/view
      insert dominated:

SCROLLING

    When the user scrolls a datamodel view (thumbView, gridView or tableView), the
    scrollbar change signal is received by thumbHasScrolled, gridHasScrolled or
    tableHasScrolled. The other views are scrolled to sync with the source view's
    midVisibleRow. The first, mid and last visible items are determined in
    MW::updateIconRange and the metaReadThread is called to update the metadata and icons
    in the range (chunk).

    However, when the program syncs the views this generates more scrollbar signals that
    would loop. This is prevented by the G::ignoreScrollSignal flag. The same flag is also
    employed in fileSelectionChange, where visible views are scrolled to centre on the
    current selection.

    Shift + navigation key (extending a selection) does not move the current index, so the
    centre-on-current scroll above does not fire. Instead, Selection::select (Shift branch)
    follows the extending selection: it scrolls the visible view(s) just enough to expose
    the cell one beyond the new selection edge, in the direction of extension (the "look-
    ahead" cell). This keeps both the last selected item and the next item visible so the
    user can see where the selection is heading. The scroll uses EnsureVisible (not centre),
    so it is a no-op when the look-ahead cell is already on screen and is clamped to the
    start/end of the datamodel. See IconView::ensureRowVisible and TableView::ensureRowVisible.

------------------------------------------------------------------------------------------
Painting Delays
------------------------------------------------------------------------------------------

    When the user changes modes in MW (i.e. from Grid to Loupe), an IconView instance
    (either thumbView or gridView) can change state from hidden to visible. Since hidden
    widgets cannot be accessed, we need to wait until the IconView becomes visible and
    fully repainted before attempting to scroll to the current index.

    Also, when an IconView resize event occurs — triggered by a change in the gridView
    cell size (thumbWidth) that requires justification, a change in the thumbDock
    splitter, or a resize of the application window — we also need to wait until
    scrollbar repainting is completed.

    The last paint event is identified by calculating the maximum of the scrollbar range
    and comparing it to the paint event, which updates the range each time. With larger
    datasets (i.e. 1500+ thumbs) it can take a number of paint events and hundreds of ms
    to complete.

    NOTE: An earlier IconView::waitUntilScrollReady helper has been removed. The wait-
    for-paint logic now lives inline in the scroll-handling paths in IconView and MW.

------------------------------------------------------------------------------------------
Scrollbar Change Flow
------------------------------------------------------------------------------------------

    • The ScrollBar signal valueChanged triggers MW::thumbHasScrolled,
      MW::gridHasScrolled or MW::tableHasScrolled.

    • MW::updateIconRange updates the first, mid and last visible icons from
      all views to find the greatest range of visible icons by calling
      IconView::updateVisible and TableView::updateVisible. dm->iconChunkSize
      is adjusted if it is less than the visible icon range.

    • Concurrent metadata loading:
        MW::update(midVisibleIcon) is called and it loads any missing icons and cleans up
        any orphaned icons (not in the iconChunkRange).

------------------------------------------------------------------------------------------
Resizing Icons or the Icon Viewport
------------------------------------------------------------------------------------------

    When the IconView viewport is resized or the icon size is changed, the result is a
    scroll event. Since we want to keep the view centred on its original state, the middle
    or centre icon is tracked as dm->scrollToIcon. G::resizingIcons is set for the resize
    and the subsequent scroll event does not reset dm->scrollToIcon. MW::thumbHasScrolled
    or MW::gridHasScrolled resets G::resizingIcons to false.

    This allows the three datamodel views to be synced using dm->scrollToIcon.

    Persisted icon size (between sessions and workspaces):
        IconView::assignedIconWidth is the reference width that rejustify() uses to
        compute thumbs-per-row and then justify the cell width to exactly fill the row.
        The justified result lands in iconWidth, which can differ from assignedIconWidth.
        Critically, iconWidth is viewport-dependent: a rejustify() at a narrow width
        shrinks iconWidth toward ICON_MIN, while assignedIconWidth is unchanged.

        Rule: persist (save) assignedIconWidth, not iconWidth. iconWidth can be corrupted
        by a transient rejustify() — at startup the hidden gridView receives resize events
        at a narrow width, each scheduling a rejustify() ~500ms later (IconView::resizeEvent)
        that justifies iconWidth down to e.g. 62. Saving iconWidth persisted that transient
        size, so the grid reverted to ~62 on the next launch; assignedIconWidth survives the
        transient. (thumbView has an extra guard: rejustify() skips "Thumbnails" while
        G::isInitializing, and MW::showEvent applies its size via setThumbSize().)

        Rule: on restore, set assignedIconWidth = iconWidth BEFORE calling rejustify().
        rejustify() justifies from assignedIconWidth, so if only iconWidth is restored the
        saved size is discarded and rejustify() uses a stale/uninitialised reference.

        Where these rules are applied:
            • MW::writeSettings / MW::createGridView  - between-sessions save/restore.
              writeSettings saves gridView->assignedIconWidth as thumbWidthGrid;
              createGridView loads it into iconWidth then copies to assignedIconWidth.
            • MW::saveWorkspace / MW::invokeWorkspace  - per-workspace save/restore for
              both thumbView and gridView (save assignedIconWidth; on restore set
              assignedIconWidth = iconWidth before rejustify()).
            • MW::defaultWorkspace  - sets assignedIconWidth = iconWidth before rejustify()
              so the hard-coded defaults (thumb 100, grid 160) are honoured.

HELP SYSTEM

    All in-app help pages are HTML files loaded by HtmlWindow (Utilities/htmlwindow.cpp).
    Every page shares a single stylesheet, Docs/help.css, which is applied automatically
    by HtmlWindow via QTextDocument::setDefaultStyleSheet. To restyle every help page at
    once, edit help.css; no other files need to change.

------------------------------------------------------------------------------------------
Architecture
------------------------------------------------------------------------------------------

    Utilities/htmlwindow.cpp        HtmlWindow widget (a QScrollArea hosting a
                                    QTextBrowser). Loads the HTML and applies
                                    Docs/help.css as the document's default stylesheet,
                                    and sets a qrc base URL on the document so relative
                                    <img src="images/..."> resolves.

    Docs/help.css                   Shared dark-theme stylesheet for every help page.
                                    Defines body, h1-h3, a, code, pre, .highlight, .warn.
                                    Some values are fallbacks: at load time HtmlWindow
                                    overrides the body font-size with the app font size
                                    (G::strFontSize), scales the h1-h3 font-sizes by
                                    G::fontSize/12, and sets the h1-h3 colours from the
                                    theme-tied G::header1/2/3Color. The static values in
                                    help.css apply only when viewing the file outside the
                                    app (e.g. VS Code Live Preview).

    Docs/_template.html             Reference template demonstrating every supported
                                    element (headings, paragraphs, span colors, link,
                                    image, lists, code block, table). Copy this when
                                    creating a new page.

    Docs/<topic>.html               One file per help topic. The <head> only needs:
                                        <link rel="stylesheet" href="help.css">
                                    Qt ignores the <link> (it already loaded help.css
                                    as the default stylesheet); VS Code's Live Preview
                                    honors it, so the editor view matches Winnow.

    Docs/images/                    Image assets used by help pages. Reference with a
                                    relative path: <img src="images/Foo.png">.

    winnow.qrc                      Every .html and image referenced from help must be
                                    listed here so it is bundled into the binary.

    HtmlWindow lifecycle            Constructor sets Qt::Window (top-level with title bar)
                                    and Qt::WA_DeleteOnClose (self-cleanup on user close).
                                    Callers pass a parent (`window()` for embedded
                                    widgets, `this` for top-level dialogs) so the help
                                    window is destroyed with its owner. MW::closeEvent
                                    also sweeps findChildren<HtmlWindow*>() as a safety
                                    net.

------------------------------------------------------------------------------------------
Adding a New Help Page
------------------------------------------------------------------------------------------

    1. Copy Docs/_template.html to Docs/<yourpage>.html.

    2. Edit in VS Code (see "VS Code Workflow" below). Keep the existing
       <link rel="stylesheet" href="help.css"> in the <head>; only add a <style> block
       if you need per-page overrides.

       OR use BBedit.

    3. Place any images in Docs/images/ and reference them with a relative path,
       e.g. <img src="images/MyDiagram.png">.

    4. Register the new files in winnow.qrc under the existing /Docs prefix:
           <file>Docs/<yourpage>.html</file>
           <file>Docs/images/<YourImage>.png</file>     (if any)

    5. Open it from C++. Pattern (mirrors Filters::howThisWorks at filters.cpp:1767):

           #include "Utilities/htmlwindow.h"

           void MyWidget::someHelp()
           {
               QRect r = QRect(mapToGlobal(geometry().topLeft()), geometry().size());
               new HtmlWindow("Winnow - My Topic",
                              ":/Docs/yourpage.html",
                              QSize(700, 600), r, window());
           }

       The fifth argument (`window()` or `this`) sets the parent so the help window
       closes with its owner. Do not store the returned pointer or delete it manually;
       WA_DeleteOnClose handles cleanup.

    6. Build. No other wiring is required.

------------------------------------------------------------------------------------------
Editing Help Formatting Globally
------------------------------------------------------------------------------------------

    Edit Docs/help.css. Every help page picks up the change on the next launch (the .css
    is loaded fresh by HtmlWindow each time a help page opens).

    Runtime overrides: HtmlWindow::applyStyle() appends rules on top of help.css so help
    text tracks the app font and theme. These win over the help.css values (later rule,
    equal specificity), so editing them in help.css alone has no effect in the app:
      • body font-size  -> app font size (G::strFontSize)
      • h1/h2/h3 font-size -> help.css design size scaled by G::fontSize/12
      • h1/h2/h3 colour -> G::header1/2/3Color (theme-tied; set in WidgetCSS::css())
    applyStyle() also re-runs on every font/theme change (HtmlWindow::refreshOpenWindows,
    called from MW), so open help windows update live. To change these, edit applyStyle()
    (font-size/colours) or the G::headerNColor values in WidgetCSS::css() (colours).

    Constraint: QTextBrowser supports only Qt's HTML 4 / CSS 2.1 subset
        https://doc.qt.io/qt-6/richtext-html-subset.html
    Avoid flexbox, grid, CSS custom properties, and HTML5 semantic tags — Qt silently
    ignores them.

    For per-page overrides, add a <style> block to that page's <head>. It cascades on
    top of help.css.

8. ACTION ENABLE/DISABLE RULES
******************************************************************************************

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    Menu/toolbar actions that act on the loaded folder, the current selection, or the set
    of picked images are enabled/disabled centrally in MW::enableSelectionDependentMenus()
    (`Main/menusandactions.cpp`).  Pick-dependent actions are additionally kept in sync by
    MW::updatePickDependentActions() (`Main/mainwindow.cpp`), which is called from the
    pick-toggle paths.

    Actions that are always valid (app/window/help/about, dock-visibility toggles,
    workspaces, preferences, open folder, color manage, combine raw+jpg, diagnostics,
    folder-tree/USB context actions handled by the rename* slots, and "Manage External
    Applications") are never gated here.

------------------------------------------------------------------------------------------
Conditions and Triggers
------------------------------------------------------------------------------------------

    Four state conditions drive enabling.  Each must have a trigger that re-runs the
    relevant update when the condition can change, otherwise actions get stuck disabled:

    • dmHasRows  (dm->rowCount() > 0 — a folder is loaded)
         Triggers: fileSelectionChange (auto-select after load), selectionChanged (on
         clear), and view-mode switches — all call enableSelectionDependentMenus().

    • hasSelection / selection count  (sel->count())
         Trigger: the selection model's selectionChanged signal is connected directly to
         enableSelectionDependentMenus() in MW::createSelection()
         (`Main/initialize.cpp`).  This is essential: extending/shrinking a selection
         with shift/ctrl does NOT change the current index, so fileSelectionChange alone
         would never re-evaluate count-dependent actions (e.g. focusStack stuck disabled
         after growing the selection to two).

    • isAnyPick  (dm->isAnyPick())
         Triggers: updatePickDependentActions() is called from every pick-toggle path in
         `Main/pick.cpp` (togglePick, togglePickUnlessRejected, togglePickMouseOverItem,
         toggleReject, updatePickFromHistory, recoverPickLog) and after ingest in
         MW::setIngested() (`Main/setandupdate.cpp`).  enableSelectionDependentMenus()
         also calls it.

    • isUpDown  (gridView or tableView visible)
         Trigger: view-mode switches call enableSelectionDependentMenus().

------------------------------------------------------------------------------------------
The Menu vs Shortcut Pitfall
------------------------------------------------------------------------------------------

    Disabling a QMenu (menu->setEnabled(false)) only greys its entry in the menu bar.  It
    does NOT disable the window-level keyboard shortcuts of its child actions, because each
    child is added to MW via addAction() with its own shortcut (e.g. rate "1", label "6",
    filter "Shift+6").  Those shortcuts keep firing with no folder loaded.

    Therefore menus whose children carry shortcuts are gated with the local setMenuEnabled()
    helper, which disables the menu AND every child action individually:
    ratingsMenu, labelsMenu, sortMenu, filterMenu, utilitiesMenu, embelExportMenu.
    The Open With submenu is iterated the same way but stays enabled so manageAppAction
    remains reachable; only its per-app actions are gated.

------------------------------------------------------------------------------------------
Disabled Shortcut Feedback
------------------------------------------------------------------------------------------

    A disabled QAction does not fire and Qt does not consume its shortcut, so the key
    falls through as a normal KeyPress.  To tell the user why nothing happened, the qApp
    event filter (MW::eventFilter, `Main/mainwindow.cpp`) intercepts KeyPress events:

      • MW::disabledActionForShortcut(seq) scans MW::actions() for an action whose
        shortcut matches the pressed key sequence.  It returns the match only if it is
        disabled AND no enabled action owns the same shortcut (an enabled action would
        already have handled the key, so there is nothing to explain).
      • MW::actionDisabledReason(a) just reads the action's "disabledReason" dynamic
        property and formats it as "<action name> — <reason>."  The reason is recorded
        when the action is gated: enableSelectionDependentMenus() uses a gate() helper
        (and setMenuEnabled() for menu children) that sets both the enabled state and the
        property; updatePickDependentActions() does the same for ingest/nextPick/prevPick.
        Because the reason lives next to the gating, the two never drift.  A disabled
        action with no property set falls back to "not available right now".
      • On a match the reason is shown via G::popup and the key is consumed (return true).

    Guards: skipped during initialization, for auto-repeat, for modifier-only presses,
    when focus is in a text-entry widget (QLineEdit/QTextEdit/QPlainTextEdit), and when
    focus is in another top-level window (fw->window() != MW) so typing is never hijacked.

    Modeless dialog shortcut guard
    ..........................................................................................
    MW's single-key shortcuts use the default Qt::WindowShortcut context, which Qt
    activates whenever the action's window is anywhere in the parent chain of the active
    window.  A modeless dialog parented to MW (e.g. the Preferences dialog) therefore does
    NOT shield these shortcuts on its own: a digit typed into a Preferences spin box would
    fire rate "1" instead of entering text.  The enabled-action case fires via Qt's shortcut
    map (QEvent::ShortcutOverride) before the KeyPress feedback handler runs, so the
    eventFilter also intercepts ShortcutOverride: when focus is in another window
    (fw->window() != MW) and the key matches an MW-owned shortcut (MW::ownsShortcut(seq)),
    it accepts the override and consumes the event, delivering the key to the focused widget
    as normal input.  This scopes suppression to MW's own shortcuts, leaving the dialog's
    own shortcuts (typically Ctrl-modified) free to fire.

------------------------------------------------------------------------------------------
Rules by Action
------------------------------------------------------------------------------------------

    Requires dmHasRows (folder loaded):
      • File: revealFile(+context), reportMetadata, copyFolderPathFromContext,
        copyImagePathFromContext, rename, saveAsFile, refreshCurrent (always available
        while a folder is loaded — needed to pick up files added externally).
      • Edit: selectAll, invertSelection, shareFiles, copyFiles, copyImage,
        deleteImages, pick, pickMouseOver, reject, pickUnlessRejected, popPickHistory,
        tokenTemplateEditor, rotateRight/Left, ratingsMenu + rate0..5,
        labelsMenu + label0..5, utilitiesMenu (mediaReadSpeed, visCmpImages).
      • Go: jump, random, and all horizontal/home/end/page navigation (+AddToSelection)
        and the scroll family except up/down.
      • Filter: filterMenu + all filter* children.
      • Sort: sortMenu + all sort* children.
      • Embellish: embelExportMenu + children, embelNew/Read/SaveTemplate.
      • View: asLoupe, asGrid, asTable, copyInfoTextToClipboard, slideShow,
        zoomTo/In/Out/Toggle, thumbsEnlarge/Shrink.

    Requires selection count >= 2:
      • asCompare (compare view), meanStack, focusStack.

    Requires isAnyPick (at least one picked image):
      • ingest, nextPick, prevPick.

    Requires a view that scrolls vertically (isUpDown — grid or table visible):
      • keyUp/Down (+AddToSelection), keyScrollUp/Down.

    Other:
      • Open With per-app actions: require selection (hasSelection); manageAppAction
        always enabled.
      • pasteFiles: enabled when the clipboard holds file URLs
        (Utilities::clipboardHasUrls()).

MEMORY CARD EJECT

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    Ejecting a removable card (USB drive / memory card) safely unmounts it from the OS.
    The platform-specific work lives in UsbUtil (`Utilities/usbutil.cpp`):

      • UsbUtil::eject(rootPath) — Windows: DISMOUNT_VOLUME then STORAGE_EJECT_MEDIA via
        DeviceIoControl; macOS: "diskutil eject <mount point>".  Returns true only on a
        real success (the Windows path reports failure if either ioctl fails).
      • UsbUtil::isEjectable(rootPath) — true only when rootPath is a mounted volume that
        is valid (ready, writable) AND removable (DRIVE_REMOVABLE/DRIVE_CDROM on Windows,
        under /Volumes/ on macOS).  Both checks require rootPath to be the volume's mount
        root, not a subfolder.

    MW::ejectUsb(path) (`Main/mainwindow.cpp`) wraps this: if the drive being ejected is
    currently loaded in the datamodel it first calls stop() and clears the FSTree
    selection (reading files from a drive after ejection would crash), then ejects and
    shows a confirmation/failure popup.

------------------------------------------------------------------------------------------
Entry Points
------------------------------------------------------------------------------------------

    Eject is reachable ONLY from the context menus of the Folders (FSTree) and Bookmarks
    panels — there is no main-menu eject action.  The context action
    (ejectActionFromContextMenu -> MW::ejectUsbFromContextMenu) ejects the drive of the
    moused-over folder (mouseOverFolderPath).  MW::renameEjectUsbMenu() sets the action's
    label ("Eject Memory Card <drive>") and enabled state from that folder as the context
    menu is built.

------------------------------------------------------------------------------------------
Drive Root Normalization
------------------------------------------------------------------------------------------

    The moused-over (or ingest source) path is usually a subfolder, e.g. a DCIM folder,
    but isEjectable/eject require the volume mount root.  Both MW::ejectUsb() and
    MW::renameEjectUsbMenu() therefore normalize with QStorageInfo(path).rootPath() before
    calling UsbUtil, so eject works (and the menu item enables) for any folder on the card,
    not only when the volume root itself is selected.

------------------------------------------------------------------------------------------
Auto-Eject After Ingest
------------------------------------------------------------------------------------------

    The ingest dialog has an "eject" checkbox bound to the persisted setting autoEjectUsb
    ("autoEjectUSB").  When set, MW::ingest() ejects the ingest source card after a
    successful FOREGROUND ingest — both the gotoIngestFolder early-return path and the
    normal path, after the copy has finished (IngestDlg runs the foreground copy
    synchronously during exec()).  The eject targets dm->primaryFolderPath(), normalized
    to the drive root by ejectUsb().

    Auto-eject is foreground-only by design.  The ingest dialog disables AND clears the
    eject checkbox whenever background ingest is selected (IngestDlg::
    on_backgroundIngestChk_stateChanged, `Dialogs/ingestdlg.cpp`), and the checkbox is
    initialized after the background checkbox so a persisted autoEjectUsb=true is
    reconciled to false at construction.  As a result autoEjectUsb is always false for a
    background ingest; MW::ingest() additionally guards the eject with !isBackgroundIngest.
    A background ingest never auto-ejects (ejecting mid-copy would fail / corrupt the
    transfer).

CHECK FOR UPDATES
******************************************************************************************

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    Winnow checks whether a newer release is available for download.  Distribution is
    plain downloads (notarized DMG on macOS, Inno Setup installer on Windows) from the
    DigitalOcean server at https://winnow.ca, so the check only NOTIFIES the user and, on
    request, opens the platform download URL in the default browser — it does not
    auto-install.  (The old Windows-only Qt `maintenancetool` path is gone.)

    Two entry points, both calling MW::checkForUpdate(bool silent) (`Main/mainwindow.cpp`):
    • Help → "Check for updates" — silent == false (loud): also reports "up to date" and
      network errors via popups, and ignores the skip-version setting.
    • Startup, from MW::showEvent — a deferred QTimer::singleShot fires checkForUpdate(true)
      4 s after the window shows, gated by the checkIfUpdate preference and skipped when
      G::isTest (no network during headless / soak runs).  Silent: it only speaks when an
      update is available and has not been skipped.

------------------------------------------------------------------------------------------
Mechanism
------------------------------------------------------------------------------------------

    checkForUpdate() does an async GET (QNetworkAccessManager, no blocking wait) of a
    small per-platform descriptor:
        macOS:   https://winnow.ca/winnow_mac/version.json
        Windows: https://winnow.ca/winnow_win/version.json
    Shape: { "version": "2.05",
             "url":     ".../current/Winnow2.05.dmg",
             "notes":   ".../winnow/versions.html" }

    MW::onUpdateCheckReply() parses the JSON and compares with QVersionNumber (NOT string
    compare).  The running version is WINNOW_VERSION (`Main/mainwindow.h`, exposed as
    versionNumber), single-sourced from CMake project(Winnow VERSION ...).  If the remote
    version is greater it shows the UpdateAppDlg dialog (`Dialogs/updateAppDlg.*`):
    • Download → MW::downloadAndOpenUpdate(url): the installer is fetched in-app (no
      browser, progress shown in G::popup) to the Downloads folder, then opened — on
      macOS `open` mounts the DMG and brings its Finder window to the front (on top of
      Winnow); on Windows the downloaded Inno Setup installer is launched.
    • Skip this version → UpdateAppDlg::skipRequested() is true; the caller stores the offered
      version in updateSkipVersion ("updateSkipVersion") so the silent startup check no
      longer prompts for it (a loud manual check still does).
    • Not now → no state change; offered again next launch.

    Preference: checkIfUpdate ("checkIfUpdate", default true) — "Check for program updates
    at startup" in the preferences dialog (`Dialogs/prefdlg.*`, updateAppChk).

------------------------------------------------------------------------------------------
Publishing version.json
------------------------------------------------------------------------------------------

    The deploy scripts publish the descriptor (see packaging/README.md):
    • macOS deploy.command promote step 7b writes/uploads winnow_mac/version.json from the
      promoted version, and curl-checks it.
    • Windows deploy.ps1 writes/uploads winnow_win/version.json when $VersionJsonTarget is
      set.  Its url points at winnow_win/current/, so the Setup.exe must be in current/.
    Build dependency: the client needs Qt6::Network (added to CMakeLists.txt).

FIND DUPLICATES

    FindDuplicatesDlg (`Dialogs/findduplicatesdlg.*`) finds duplicates of a set of
    candidate images among a set of target images. Launched from MW::findDuplicates,
    which runs it modally (exec) and, on accept, rebuilds the Compare filter so the
    flagged duplicates can be shown.

    Two columns of data drive everything:
    • The candidate "A" list — the images being checked. Held in the datamodel proxy
      dm->sf, addressed through the aSfRows mapping (see below).
    • The target "B" list — bItems, built from the include/exclude folders dragged into
      the dialog. Each B entry caches name, type, createdDate, aspect, duration and a
      thumbnail (im) for pixel comparison.

------------------------------------------------------------------------------------------
Candidates (A list) and the model mapping
------------------------------------------------------------------------------------------

    Candidates are the SELECTED images in dm->sf, not all filtered images. setupModel()
    builds aSfRows, a list mapping each candidate index (the loop variable 'a' / the row
    in the dialog's QStandardItemModel) to its dm->sf proxy row:
    • aSfRows is populated from dm->isSelected(sfRow) over the filtered proxy.
    • If nothing is selected it falls back to all filtered rows, so the dialog is never
      empty.

    Because the candidate list is no longer 1:1 with dm->sf, every candidate access to
    the datamodel goes through aSfRow(a) (returns -1 when out of range, so an invalid
    QModelIndex yields safe empty data rather than throwing). The dialog model row and
    results[a]/matches[a] stay keyed by the candidate index a; dm->sf is reached only via
    aSfRow(a).

------------------------------------------------------------------------------------------
Targets (B list)
------------------------------------------------------------------------------------------

    buildBList() expands the include folders (optionally recursing subfolders), subtracts
    the exclude folders, then appends a bItem per supported file found.
    • If no candidate is a video (candidatesHaveVideo() is false, tested via
      G::VideoColumn), videos are excluded from the targets too — there is no point
      comparing against video duplicates when no candidate is a video.
    getMetadataBItems() then fills each bItem's metadata and thumbnail.

------------------------------------------------------------------------------------------
Comparison and thumbnail normalization
------------------------------------------------------------------------------------------

    on_compareBtn_clicked() runs the search: clear → buildBList → getMetadataBItems →
    pixelCompare() (when "same pixels" is checked) or findMatches() (criteria match:
    name/type/date/aspect/duration). matched candidates are checked in the dialog table.

    Pixel comparison only makes sense if the two thumbnails share dimensions, so both the
    candidate (A, from the decoration-role icon) and target (B) thumbnails are run through
    normalizeBThumb(): scaled to 256 px on the long side (aspect preserved, the same
    256 box used to extract the A icon) and converted to QImage::Format_RGB32. For
    matching-aspect images this yields identical dimensions.

    compareRGB() sums the per-channel absolute differences over the overlapping
    rectangle. It reads pixels directly from constScanLine() as QRgb (both inputs are
    already RGB32) — QImage::pixelColor() per pixel was ~50x slower and, run over every
    A x B combination, was the original cause of the long main-thread stall (beachball).

------------------------------------------------------------------------------------------
Video frames (async sync)
------------------------------------------------------------------------------------------

    Video target thumbnails come from FrameDecoder asynchronously: getMetadataBItems()
    queues each video (addToQueue) and increments pendingVideoFrames; the first frame
    arrives later at setImageFromVideoFrame(), which stores the normalized thumbnail and
    decrements the counter. getMetadataBItems() then WAITS for pendingVideoFrames to
    reach 0 before returning (pumping the event loop, with a 30 s timeout and abort
    check), so pixelCompare() never runs against null video thumbnails. pendingVideoFrames
    is reset in clear().

------------------------------------------------------------------------------------------
Responsiveness
------------------------------------------------------------------------------------------

    The whole search runs synchronously on the GUI thread. The dialog's progress/abort
    hooks were originally gated on G::useProcessEvents, which is globally false, so the
    GUI froze for the entire run. The compute loops (getMetadataBItems, pixelCompare,
    findMatches) now pump the event loop unconditionally; pixelCompare throttles to one
    update/processEvents per percent to avoid flooding the loop. on_compareBtn_clicked
    guards re-entry (if (isRunning) return), disables compareBtn during the run, and
    resets isRunning / re-enables the button on every exit path.

------------------------------------------------------------------------------------------
Output
------------------------------------------------------------------------------------------

    on_updateDupsAndQuitBtn_clicked() writes the result back to the datamodel: it first
    clears G::CompareColumn on ALL dm->sf rows (candidates are only a subset, so a
    candidate-only pass would leave stale flags), then sets it true for each checked
    candidate via aSfRow(a), and accepts the dialog. MW::findDuplicates then refreshes
    the Compare filter.

------------------------------------------------------------------------------------------
Dialog layout
------------------------------------------------------------------------------------------

    findduplicatesdlg.ui lays the upper area out as leftFrame then rightFrame inside
    horizontalLayout_13. leftFrame (on the left) holds the candidate images table (tv);
    rightFrame (on the right) holds the target include/exclude folder lists and the
    comparison controls. The frame names reflect their on-screen positions.

RAW DECODING (SENSOR DEMOSAIC)
==========================================================================================

------------------------------------------------------------------------------------------
Architecture
------------------------------------------------------------------------------------------

    Full-sensor RAW decoding (demosaic), based on LibRaw, is being ported in phases. It is
    separate from the existing embedded-JPG preview path: a RAW file still shows its
    embedded JPG until a per-format sensor decoder exists.

    Pipeline (ImageFormats/Raw/):
        ImageDecoder::load()
          -> RawFormat::Create(ext)          factory; "arw" -> SonyRaw, else nullptr (embedded JPG)
          -> RawFormat::Decode()             shared, camera-agnostic pipeline:
               1. UnpackCfa()                 PER-FORMAT virtual (THE override point):
                                              vendor packed/compressed bitstream -> normalised
                                              CFA mosaic (RawImage). Lives in the vendor .cpp
                                              (canon.cpp, nikon.cpp ...) as a class SEPARATE
                                              from that file's metadata parser.
               2. SubtractBlack()            shared per-2x2-position black subtraction + clip.
               3. Demosaic::Run()            shared, camera-agnostic: Bilinear3x3 for Bayer,
                                              XTransWindow (5x5) for Fuji X-Trans; AHD/DCB later.
                                              Operates only on the CFA mosaic + pattern (Bayer) or
                                              raw.xtrans (6x6 map), so it is unit-testable.
               4. RawColor::ToWorking()      shared white balance + camera->sRGB matrix, STOPS
                                              AT LINEAR (no gamma) -> WorkingImage. Real colour
                                              (dcraw cam_xyz_coeff); see COLOUR below.
               5. Develop::Apply()           parametric edits in linear space, applied here for
                                              RAW (native float) when EditParams is non-identity.
               6. OutputTransform::ToImage() baseline tone curve (RAW only) + gamma -> QImage.

    Demosaic is deliberately NOT the override point: the algorithm is shared across cameras;
    only the unpack/decompress step is vendor-specific (mirrors LibRaw's real structure).

    Threading: one RawFormat instance is constructed per decode (per decoder thread). Unlike
    the Metadata-owned parser singletons, RawFormat instances are never shared, so the
    per-decode scratch carries no cross-thread races.

    SonyRaw::UnpackCfa (ImageFormats/Sony/sony.cpp) handles UNCOMPRESSED ARW (Compression 1,
    e.g. A9 II: 14-bit LE uint16, Bayer RGGB) AND lossy "ARW Compressed" (Compression 32767):
        UNCOMPRESSED: reads StripOffsets/Counts, ImageWidth/Length, BitsPerSample, CFAPattern
            (33422), WhiteLevel (50717), via the plumbed RawSensorInfo or a self-walk.
        ARW2 COMPRESSED (decodeSonyArw2, tried first): each 16-byte group encodes 16 photosites
            of one Bayer colour across 32 columns -- an 11-bit max+min, the indices holding them,
            then 7-bit deltas shifted to fit (max-min). The 11-bit values expand through a Sony
            tone curve from tag 0x7010 (4 control points): breakpoints at point>>2, slope starts
            at 4 and doubles each segment; sample = curve[pix<<1]>>2. Columns are de-interleaved
            even-then-odd within each 32-col group. Ported from dcraw's sony_arw2_load_raw and
            VERIFIED byte-identical to libraw on an A9 ARW. Levels/WB/matrix from the same SR2
            tags as the uncompressed path. The NEWER "Lossless Compressed RAW" (Compression JPEG,
            e.g. A1/A7R5) is lossless JPEG -> can reuse LosslessJpeg later; not handled yet.
    Other comp values -> Decode() returns false -> ImageDecoder falls back to the embedded JPG.

    METADATA PLUMBING (RawSensorInfo): the per-file facts UnpackCfa needs (CFA strip
    offset/length, geometry, bitsPerSample, compression, CFA pattern, levels, sample byte
    order) live in struct RawSensorInfo (Metadata/rawsensorinfo.h -- dependency-free), carried
    on ImageMetadata::rawInfo. Sony::parse fills it once during the normal metadata read (from
    the SubIFD0 it already parses -- no second walk). Persistence to the cache decode path:
    DataModel keeps a lock-guarded QHash<fPath, RawSensorInfo> (fPathRawInfo*), written in
    addMetadataForItem and cleared on folder reload; ImageDecoder fetches it via
    dm->fPathRawInfoGet() in cache mode (indMeta carries it directly in independent mode) and
    hands it to RawFormat::Decode.

    GATED ON G::useRaw: when raw decoding is OFF (previews only), both the parse-time fill and
    the addMetadataForItem store are skipped, so the MetaRead path pays nothing -- important
    because the parse-time extraction grows (makernote colour matrix etc.). If useRaw is toggled
    on after a folder is read, rawInfo will be absent from the store and UnpackCfa falls back to
    its self-walk, so correctness holds; the store repopulates on the next folder read.
    SonyRaw::UnpackCfa uses rawInfo when valid and otherwise falls back to its self-contained
    TIFF/EP walk (readSonyTiffSensorInfo). CRUCIAL: the self-walk fills the SAME fields as the
    parse path -- geometry AND colour (model->matrix via tag 272, WB_RGGBLevels 0x7313, BlackLevel
    0x7310). If it only filled geometry, a toggle-on after a useRaw-off load would render with an
    identity matrix / no WB (green tint). Both paths therefore produce identical colour. The
    CFAPattern->enum mapping is shared via cfaPatternFromPlaneColor() in rawimage.h.

    COLOUR (RawColor::ToWorking, real): follows dcraw/libraw cam_xyz_coeff. Input is the model's
    XYZ(D65)->camera matrix (DNG ColorMatrix / adobe_coeff convention), looked up by model in
    ImageFormats/Raw/cameramatrix.cpp -- a FULL port of libraw's adobe_coeff for the mainstream
    still-camera makers (776 models; phone/action/cinema/bare-sensor and 4-colour CMYG/RGBE
    entries omitted). Each row also carries adobe_coeff t_black/t_maximum, read via
    cameraLevelsForModel() and used as a fallback black/white where a decoder would otherwise
    guess (e.g. Canon, whose saturation is below the bit-depth max -- 7D Mark II clips at 13584,
    not 16383). Lookup is a case-insensitive longest-prefix match, so one "Nikon D800" row covers
    D800/D800E and "Sony ILCE-9" covers ILCE-9/9M2, while the most specific key wins (a short
    "Nikon D3" never shadows "D300"). Regenerate from a newer libraw colordata.cpp via
    tools/gen_cameramatrix.py when bodies are added; a wrong matrix is a hue shift, not a crash,
    so verify visually. From it RawColor
    builds camRgb = xyzToCam.rgbToXyz, row-normalises so neutral maps to neutral, derives the
    neutral white balance preMul = 1/rowSum, and inverts to get camera->linear-sRGB (rgbCam).
    It applies the white balance (as-shot raw.camMul when the parser filled it, else preMul;
    normalised to green) and the matrix, clamps to [0,1], stays linear. The matrix + camMul ride
    on RawSensorInfo (xyzToCam, camMul, hasColorMatrix), filled at parse time (model known) so
    they plumb to the cache path like the rest of rawInfo. An unknown model -> identity xyzToCam
    -> sRGB-assumed pass-through (approx, still renders).

    SONY AS-SHOT WB + BLACK: Sony::parse reads the as-shot white balance and black level from
    the SubIFD0 it already parses (the "SR2" tags, stored in the clear on ARW): WB_RGGBLevels
    (0x7313, SSHORT x4, R G1 G2 B) -> rawInfo.camMul (R,G,B,G2, applied directly as gains),
    BlackLevel (0x7310, SHORT x4) -> rawInfo.black, WhiteLevel (0xC61D) -> rawInfo.white. So
    RawColor uses the real as-shot WB (haveAsShot path), not the matrix-derived neutral, on the
    A9 II. (Found by instrumenting the file; exiftool was used only as a reference oracle.)
    LIMITATION: clipped highlights can show a slight tint until highlight recovery lands.

    ADOBE DNG (DngRaw, ImageFormats/Dng/dng.cpp): self-contained -- DngRaw::UnpackCfa walks the
    DNG's TIFF/EP IFDs (via the shared TiffWalk reader), finds the CFA image (Photometric ==
    CFA 32803, full-res, most pixels), and decodes it. Handles UNCOMPRESSED (Compression 1) and
    LOSSLESS-JPEG (Compression 7), in STRIP or TILE layout (tiles placed left-to-right,
    top-to-bottom, edge-cropped). DNG carries its OWN colour, so DngRaw reads it from the file:
    xyzToCam = ColorMatrix2 (D65; else ColorMatrix1), WB camMul = 1 / AsShotNeutral, plus the
    DNG BlackLevel (50714) / WhiteLevel (50717) / CFAPattern (33422) -- no per-model table and
    NOTHING plumbed at parse time (the decode is on-demand, so there is no MetaRead cost; it
    always self-walks). Registered for "dng" in RawFormat::Create. VERIFIED: C++ CFA output is
    byte-identical to the validated reference on a single-strip lossless DNG (Leica M10) and a
    726-tile lossless DNG (Nikon D850 / Lightroom), both of which render to coherent, correctly
    coloured images. LIMITATION: assumes ColorMatrix2's illuminant is D65 (no chromatic
    adaptation yet); packed sub-16-bit uncompressed storage not handled (16-bit storage only).

    FUJIFILM RAF (FujiRaw, ImageFormats/Fuji/fuji.cpp): RAF is a Fuji container ("FUJIFILMCCD-RAW "):
    a big-endian header with offsets (92 = CFA metadata dir, 100/104 = CFA data offset/length) to
    an embedded JPEG, a CFA directory (tag 0x100 = raw H/W; tag 0x131 = the X-Trans 6x6 colour map,
    36 bytes REVERSED -> [yy][xx], 0=R 1=G 2=B; tag 0xC000 = RAFData block with the WB), and the
    CFA data. The FujiIFD that prefixes the CFA block (cfaOff -> little-endian TIFF -> 0xf000 ->
    FujiIFD) carries BitsPerSample (0xf003 -> white), per-CFA BlackLevel (0xf00a) and the
    compressed-strip offset (0xf007). Two CFA encodings, routed on the 16-byte header at cfaOff +
    0xf007 (signature 0x4953 = compressed):
      - UNCOMPRESSED: the CFA block is rawW*rawH*2 bytes plus a small leading header (size =
        cfaLen - rawW*rawH*2, e.g. 2048), little-endian 16-bit samples.
      - COMPRESSED (lossless/lossy, most current bodies): FujiCompressed::decode
        (ImageFormats/Fuji/fujicompressed.cpp) -- a faithful port of libraw's fuji_compressed.cpp
        with all decoder state in a per-call Ctx struct (no globals, so concurrent cache-thread
        decodes are safe). The 16-byte BE header gives lossless flag, raw_type (16=X-Trans,
        0=Bayer), bits, geometry, block_size(0x300)/blocks_in_row/total_lines; each vertical
        block is a gradient-predicted, adaptive Golomb-Rice bitstream decoded into the rawW*rawH
        mosaic. Fuji Bayer bodies (GFX) are RGGB; X-Trans uses the 0x131 map.
    Active-area crop: the decoded mosaic is the FULL sensor (incl. optical-black margins, ~190
    black columns on the right of a GFX); crop to RawImageCropTopLeft (0x0110 = top,left) +
    RawImageCroppedSize (0x0111 = H,W) to drop them, else they show as a black edge. Bayer keeps an
    even origin (RGGB phase); X-Trans rolls the 6x6 map by (top%6,left%6) so the demosaicer, which
    tiles from (0,0), still gets the right colours.
    WB: the 0x2FF0 tag when present, else the as-shot WB_GRBLevels [G,R,B] in the 0xC000 RAFData
    block. That block ALSO holds WB presets (daylight/tungsten/...) that look identical to a WB
    triple, so a plain first-match scan can grab the tungsten preset and badly over-saturate; the
    as-shot WB is instead written as THREE identical consecutive triples, so we require a 3x repeat
    (G = green reference, smallest). Robust across bodies without libraw's per-model wb_section
    offset table; within ~1% of libraw on GFX, ~3% on X-T2 (libraw additionally does two-record CCT
    averaging -- visually negligible for the preview/develop path).
    Matrix by model (RAF header camera id at 0x1C -> "Fujifilm X-T50"). X-Trans is demosaiced by
    Demosaic::XTransWindow (5x5 same-colour average over the tiled 6x6 map; native channel exact --
    the X-Trans analogue of the bilinear Bayer path). VERIFIED byte-identical to libraw on the
    uncompressed X-T50 and the compressed X-Trans (X-T2) and Bayer (GFX 50S II) RAFs. LIMITATIONS:
    the lossy compressed path is ported but unexercised (no lossy test file); white level is the
    FujiIFD BitsPerSample max (e.g. 16383 at 14-bit) not libraw's tighter per-body saturation
    (e.g. GFX 15872), so highlights clip slightly late; WB is unaveraged (~3% on X-T2).

    PANASONIC RW2 (PanasonicRaw, ImageFormats/Panasonic/panasonic.cpp): RW2 is a TIFF-like file
    (magic 0x55) whose IFD0 carries Panasonic raw tags -- SensorWidth/Height (0x02/0x03),
    RawFormat version (0x2D), RawDataOffset (0x118), CFA (0x09: 1=RGGB 2=GRBG 3=GBRG 4=BGGR),
    per-channel WB (0x24/0x25/0x26 = R/G/B, green==256) and black (0x1C..0x1E, stored 15 below
    true). Decodes the classic RawFormat-4 compression (dcraw panasonic_load_raw): a 14-column
    group code with a per-pair nibble predictor, bits pulled from Panasonic's reversed 16 KB
    "pana_bits" buffer (refilled rotated by loadFlags = 0x2008; byte index vbits>>3 ^ 0x3FF0).
    Ported from dcraw, validated BYTE-IDENTICAL to libraw on a DC-GX9 RW2; renders to a coherent,
    correctly-coloured image. Matrix from the per-model table (DC-GX9). LIMITATION: RawFormat
    5/6/7 (newer cameras) use different loadFlags / a different codec -> returns false (fallback).

    OLYMPUS ORF (OlympusRaw, ImageFormats/Olympus/olympus.cpp): the ORF TIFF LIES (tags the raw
    strip Compression 1 / 16-bit) but the strip is Olympus's proprietary 12-bit compression: a
    Huffman magnitude code (a direct 4096-entry 12-bit-prefix table) + an adaptive bit length per
    sample + a median predictor (w,n,nw) with a 2-column running carry; the bitstream starts 7
    bytes into the strip. Ported from dcraw's olympus_load_raw, validated byte-identical to libraw
    on an E-M1 ORF. As-shot WB is read from the Olympus MakerNote (non-standard "OLYMPUS\0II\3\0"
    header -> TiffWalk::initEmbedded with base = makernote start, IFD at +12; ImageProcessing
    0x2040 -> WB_RBLevels 0x0100, two SHORTs R,B with green == 256) -- essential, matrix-neutral
    renders badly green for Olympus. Matrix from the per-model table (E-M1). FIRST CUT: white 4095
    (12-bit), E-M1-validated black 255 + BGGR pattern (ORF has no CFAPattern tag); per-model
    black/pattern/bit-depth is a refinement (the OM-1 may differ). NOTE: TiffWalk::init now
    accepts non-42 TIFF magics (ORF 0x4F52, RW2 0x55, CR2 0x4352) since the endian marker + a sane
    first-IFD offset is enough; standard TIFF (magic 42) still validates.

    NIKON NEF (NikonRaw, ImageFormats/Nikon/nikon.cpp): self-contained. Walks the NEF IFDs (via
    TiffWalk) to the CFA SubIFD (Compression 34713 = Nikon NEF Compressed) -- some bodies (Z9)
    expose SEVERAL such SubIFDs (the full-res frame plus smaller/0x0 placeholders), so the largest
    valid-geometry candidate is chosen, not the last seen (else the 0x0 IFD fails the geometry check
    and the whole decode falls back to the embedded preview), then reads the
    linearization/Huffman metadata from the Nikon type-3 MakerNote (tag 0x96). The MakerNote is
    an EMBEDDED TIFF ("Nikon\0" + 2 version bytes + "\0\0" + its own header at base = mnOffset+10,
    offsets relative to base) -- TiffWalk::init(file, base) handles that. Nikon's compression is
    NOT lossless JPEG: each sample is a Huffman-coded difference (one of dcraw's 6 trees, chosen
    by version byte + bit depth) against DUAL predictors -- a vertical predictor for the first two
    columns (per row parity), then a running horizontal predictor per column parity -- mapped
    through a linearization curve (identity for the 0x46 lossless mode; interpolated with a tree
    "split" row for the 0x44 lossy mode). Ported from dcraw's nikon_load_raw. Colour matrix from
    the shared per-model table (added D810/D7200/D850/D2H/D100/D800E/Z9; an unknown model falls back
    to an identity matrix, which renders with a colour cast -- seen on the Z9 before its row was
    added); white = (1<<bps)-1; black from MakerNote
    tag 0x3d (4-SHORT per-channel pedestal: D850 400, D810 600, Z9 1008 -- matches libraw), falling
    back to the decoded minimum only on older bodies that omit 0x3d (D2H/D100/D800E, ~0 pedestal).
    The minimum alone is fragile: one sub-pedestal hot/noisy pixel drags it below the true black, so
    the shared SubtractBlack under-subtracts and the frame renders washed-out (was seen on the D850,
    darkest pixel 78 vs true 400). CFA phase read from the raw SubIFD's
    CFAPattern tag (0x828E) via cfaPatternFromPlaneColor() -- most bodies are RGGB but older
    sensors differ (D100 GRBG, D2H GBRG), and decoding those as RGGB rendered greens as magenta;
    falls back to RGGB when the tag is absent. ACTIVE-AREA CROP: most NEFs store exactly the active
    area, but a few old bodies pad it with masked optical-black columns that fringe the left/right
    edges magenta when demosaiced; a per-model crop (libraw margins -- D2H: left 6, width 2482)
    trims them. Margins are even so the CFA phase is unchanged; add a row per affected model.
    VERIFIED: the C++ decode is
    BYTE-IDENTICAL to libraw on 14-bit (D810) and 12-bit (D7200) lossless NEF -- validated with a
    throwaway rawpy/libraw reference in scratch, since the bit-level scheme is too subtle to
    guess. As-shot WB: read from the UNENCRYPTED MakerNote tag 0x0C (WhiteBalanceRBLevels, 4
    RATIONALs in order R, B, G1, G2 -- green-normalised multipliers) -> raw.camMul, matching
    libraw's cam_mul exactly on D810/D7200/D100. (Some bodies, e.g. D100, store only R/B and leave
    the green entries 0, meaning green=1; substitute 1.0 rather than rejecting the tag, else a zero
    green multiplier drops the WB and leaves a heavy green cast.) Old bodies that lack 0x0C (e.g. D2H) instead carry
    WB in the ColorBalance block 0x97; its early pre-encryption versions ("0100".."0103") store the
    same RGGB levels as 16-bit ints at a fixed, version-dependent offset (layout ported from dcraw's
    parse_makernote) -- reproduces libraw's cam_mul on D2H (611,256,397,256). Without it the matrix-
    derived neutral WB was used, giving a warm cast. (The version-0223+ cipher is still unbroken but
    those bodies all expose 0x0C, so it is unneeded.) LIMITATION: the 0x44 lossy-with-curve path is
    ported but not yet exercised by a test file.

    CANON CR2 (CanonRaw, ImageFormats/Canon/canon.cpp): self-contained. Walks the CR2 IFD chain
    (via TiffWalk), finds the raw IFD (the one with Canon slice tag 0xC640), decodes its
    lossless-JPEG strip (Compression 6), then reassembles Canon's vertical SLICES: tag 0xC640 =
    [nFullSlices, sliceWidth, lastSliceWidth]; the component-interleaved JPEG raster maps back to
    sensor (row,col) by dcraw's formula. It then reads the Canon makernote (IFD0 -> ExifIFD
    0x8769 -> MakerNote 0x927C) for SensorInfo (0xE0 -> active-area crop L/T/R/B) and ColorData
    (0x4001 -> WB_RGGBLevelsAsShot; offset 63 for modern ColorData, 25/24/142 for older lengths).
    Black is self-calibrated from the masked optical-black columns (x < left) before cropping;
    white = (1<<precision)-1; colour matrix from the shared per-model table (added 7D Mark II);
    pattern RGGB; crop origin forced even to keep the Bayer phase. VERIFIED: C++ un-sliced CFA is
    byte-identical to the validated reference on a 7D Mark II CR2 (which renders to a coherent,
    correctly-coloured image), and the makernote crop/WB/model read back correctly. LIMITATIONS:
    lossless-JPEG CR2 only (sRAW/mRAW and very old CR2 layouts not handled); Canon black is a
    masked-column average, not per-channel; ColorData WB offset table covers the common versions.

    TiffWalk (ImageFormats/Raw/tiffwalk.*): a tiny shared TIFF/EP IFD reader (header, IFDs +
    SubIFDs, tag value accessors: scalar / u32 array / real array / bytes / ascii, handling the
    inline-vs-offset rule and endianness). DngRaw uses it; other vendor unpackers can adopt it
    to stop re-implementing the IFD-walk boilerplate (SonyRaw still has its own inline walk).

    SHARED LOSSLESS-JPEG (ImageFormats/Raw/losslessjpeg.*): the compression behind Canon CR2,
    most compressed Adobe DNG, and lossless NEF/ARW. NOT DCT JPEG -- the lossless mode (SOF3):
    each sample is Huffman-coded as the difference from a spatial predictor of its neighbours.
    LosslessJpeg::Decode(bytes) -> Image{width,height,components,precision,samples} with samples
    interleaved by component (samples[(y*width+x)*comps + c]). It is camera-agnostic; each vendor
    decoder extracts the bytestream from its container and reassembles components/tiles/slices
    into the CFA (that mapping is format-specific, lives in the caller). SCOPE: single scan,
    Huffman, predictors 1-7, point transform; NO restart intervals yet (returns false on DRI>0).
    VERIFIED: the C++ output is byte-identical to a reference decoder that was validated
    end-to-end on a real lossless-JPEG DNG (Leica M10: 14-bit, 2 components, predictor 1) by
    demosaicing/colour-converting it to a coherent, correctly-coloured image.

    Phasing: 1 scaffold (done) -> 2 first uncompressed format UnpackCfa: Sony ARW (done)
    -> 3 real colour: per-model matrix + matrix-derived WB + Sony as-shot WB/black (done); TODO
    more model matrices -> 4 better demosaic (AHD) -> 5 compression: shared lossless-JPEG decoder
    (done); wire into DNG/CR2 UnpackCfa next; compressed NEF/ARW later.

    NOTE: an earlier (2025) unused scaffold (ImageFormats/Base/rawbase.* +
    ImageFormats/Tiff/tiffdecoder.*) used the rejected demosaic()-as-override axis and has
    been removed.

    NOTE: RawColor now stops at LINEAR working-space RGB (no tone/gamma/ICC baked in); the
    display conversion is owned by the shared OutputTransform stage (see Develop section).


==========================================================================================
DEVELOP / IMAGE EDIT (PARAMETRIC ADJUSTMENTS)
==========================================================================================

------------------------------------------------------------------------------------------
Architecture
------------------------------------------------------------------------------------------

    Lightroom-style non-destructive adjustments (denoise, white balance, exposure, contrast,
    highlights, shadows, whites, blacks, texture, dehaze ...). They apply to ALL image types
    (JPEG/TIFF/HEIC/RAW), so they are a shared stage AFTER decode and BEFORE the display
    tone/gamma/ICC transform -- NOT part of RawFormat.

    Decisions (settled): adjustments run on a FLOAT LINEAR working buffer (exposure/tone/
    dehaze need headroom), and apply to ALL image types (not raw-only).

    Unifying representation -- WorkingImage (float linear scene-referred RGB + working colour
    space + white reference). Both decode paths converge on it:
        RAW:      ... -> Demosaic -> camera-RGB -> working space   (RawColor stops at LINEAR)
        NON-RAW:  QImage decode -> un-gamma to linear -> working space
    Consumer:
        WorkingImage -> Develop(EditParams) -> OutputTransform(tone+gamma+ICC) -> QImage

    EditParams -- one versioned struct, identity defaults (absent = no-op = no behaviour
    change). The single source of truth for all adjustment values; every stage reads from it.
    Persisted per-image (datamodel column + sidecar; deferred). The slider UI binds to it.

    Develop -- stateless/reentrant processor, constructed per decode (same discipline as
    Demosaic/RawFormat). FIXED, hard-coded operation order:
        1. Denoise (luma/chroma)        earliest; for RAW ideally on the CFA mosaic
        2. White balance                linear channel scale (RAW: camMul, from temp/tint)
        3. Exposure                     linear gain 2^EV
        4. Contrast                     S-curve about mid-grey
        5. Highlights/Shadows/          luminance-masked parametric tone
           Whites/Blacks
        5b. Colour RGB                  per-channel linear gain (red/green/blue), folded into #2
        5c. Colour HSL                  hue rotation + saturation + luminance, after the tone curve
        6. Texture                      mid-frequency local contrast
        7. Dehaze                       local contrast + black-point, haze estimate
        (room left for Vibrance/Clarity/Sharpen)

    Two ops are special for RAW: white balance and denoise give better results applied inside
    the raw pipeline (WB at the camera-RGB stage; denoise on the CFA mosaic) rather than
    generically post-demosaic. EditParams is shared, so RawFormat::Decode() takes an optional
    const EditParams* and consumes those two fields at the right stages; Develop handles the
    rest. (Scaffold may start with everything in Develop and push those two down later -- a
    quality-only refactor, no interface change.)

    WIRING (done): the decode path now runs through these classes.
        NON-RAW: ImageDecoder::load() decodes to QImage as before; then ImageDecoder::
                 applyDevelop() (after rotate(), before colorManage()) does
                 InputTransform -> Develop -> OutputTransform. It is a NO-OP and is skipped
                 entirely when EditParams is identity, so the unedited browsing path is
                 unchanged and never round-trips through the float buffer. When it does run it
                 goes through the WorkingImage cache (reuses / populates the pre-develop image).
        RAW:     RawFormat::Decode() develops in its native linear float (RawColor::ToWorking
                 -> Develop -> OutputTransform), hands its pre-develop WorkingImage back via
                 outWork (cached by ImageDecoder), and ImageDecoder sets developApplied so
                 applyDevelop() does not run a second time. See PERF / WORKINGIMAGE CACHE below.

    ABORT: RawFormat::Decode() takes the decoder's abort atomic and polls it between stages and
        once per row inside the demosaic loop, returning false (lastError "Aborted") if set.
        This matters because ImageCache::stop() (quit) uses a BlockingQueuedConnection to each
        decoder; without polling, an in-flight 24MP demosaic would run to completion while the
        main thread blocks waiting -> beachball. ImageCache::stop() now also calls
        decoders[id]->abortProcessing() (sets the atomic, non-blocking) BEFORE that blocking
        wait so the in-flight decode actually sees the flag and bails.

    G::useRaw TOGGLE: ImageDecoder::load() only attempts the RAW path when G::useRaw is true;
        when false it skips RawFormat entirely and shows the embedded preview/JPG (prior
        behaviour). A status bar button (MW::useRawBtn, between reverseSortBtn and
        panToFocusToggleBtn) toggles it -- colored raw.png icon when on, grayscale raw_bw.png
        when off. MW::toggleUseRaw rebuilds the full-size image cache so the change is visible
        immediately. The value persists across sessions via QSettings ("useRaw", default
        false). NOTE: the raw decode is still also gated by RawFormat::Create() returning a
        decoder for the extension (arw/dng/cr2/nef/orf/rw2/raf today; others -> embedded JPG).
        Saved Develop edits are also gated on useRaw: MW::applyDevelopPreviewIfEdited() overlays
        a non-identity EditStack onto the loupe only when useRaw is on, since the edits are
        calibrated against the demosaiced RAW render. In preview mode (useRaw off) the embedded
        JPG is shown untouched; toggling useRaw on rebuilds the cache and the developed look returns.
    InputTransform (Develop/) is the non-raw "un-gamma to linear" step, the symmetric
    counterpart of OutputTransform.

    BASELINE TONE CURVE: a scene-referred raw render is radiometrically linear and lands dark
    and flat next to a camera JPEG (measured ~+0.7 EV / much less contrast on the A9 II). So
    WorkingImage carries a sceneReferred flag: RawColor sets it true (sensor data, with
    highlight headroom -- RawColor now clamps only negatives, not >1), InputTransform sets it
    false (a JPEG already has the camera tone curve baked in). OutputTransform applies a fixed
    baseline tone curve ONLY when sceneReferred: exposure x kBaselineExposure (=1.6, ~+0.68 EV)
    then the ACES (Narkowicz) filmic shoulder, before the sRGB transfer. This lifts midtones and
    rolls highlights off smoothly (vs hard-clipping), landing close to the embedded preview
    (mean luma 0.434 vs 0.445; validated in numpy against the preview). It is a fixed default
    look, not per-image auto-exposure; a future Develop "exposure" can override it. Non-raw
    output is unchanged (tone curve skipped), so the unedited-JPEG round trip stays identity.

    PERF (keeps Winnow fast): WORKINGIMAGE CACHE (Develop/workingimagecache.h, DONE). The
    expensive part of a RAW is UnpackCfa -> Demosaic -> RawColor, which produces the pre-develop
    WorkingImage; Develop + OutputTransform that follow are cheap. So that WorkingImage is cached
    (keyed by absolute file path) and an interactive slider change re-renders via only
    Develop + OutputTransform -- never decode or demosaic again.

        Storage: WorkingImageCache::instance().put(fPath, work). RAW caches the pre-develop
            WorkingImage that RawFormat::Decode() now hands back through its outWork out-param
            (a shared_ptr -- no pixel copy; the snapshot is always pre-develop, so a non-identity
            edit develops on a private copy and leaves the cached image pristine). Non-raw caches
            the post-InputTransform WorkingImage the first time applyDevelop() builds one.
        Re-render: WorkingImageCache::get(fPath) + WorkingImageCache::render(work, edit, out).
            render() is static/stateless: identity edit -> OutputTransform only (no copy);
            non-identity -> copy + Develop + OutputTransform. ImageDecoder::applyDevelop() now
            uses this fast path (cache hit skips InputTransform; miss builds + caches once).
            The editor (slider UI, deferred) calls get()+render() directly to re-render without
            a decode; a cache miss falls back to a full decode (which re-populates the entry).
        Memory: entries are shared_ptr<const WorkingImage> (large -- ~4x the 8-bit QImage), so
            the cache is byte-budgeted (default 768 MB, setMaxBytes()) with LRU eviction; the
            most-recently-used entry is never evicted (the active edit always hits). It is a
            small edit accelerator for recently-viewed images, NOT a second image cache.
            ImageCache::initialize() clear()s it on every folder change.
        Done (2026-06): downscaled proxy for live drag + full-res settle render run async off the
            GUI thread (see "Load Responsiveness" / renderDevelopFullResAsync).
        Still TODO: explicit pin of the active image so heavy ahead-caching can't evict it before
            the user edits.

    Phasing: 1 WorkingImage + split RawColor at linear + InputTransform/OutputTransform +
    Develop skeleton (identity), wired no-op through decode (DONE) + WorkingImage cache hook
    (DONE) -> 2 implement ops (exposure/WB/contrast/tone first; texture/dehaze/denoise later)
    -> 3 push RAW white balance/denoise pre-demosaic for quality -> 4 persistence + slider UI.
    NOTE: the single-EditParams render above is the SCAFFOLD. The real editor generalises it
    to a LAYER + MASK stack (next subsection); EditParams stays the per-layer unit, render()
    grows to take an EditStack, and the Develop op math is unchanged.

------------------------------------------------------------------------------------------
Layer & masking model (planned)
------------------------------------------------------------------------------------------

    GOAL. Apply any edit either globally or through one or more masks, with the SAME edit set
    shared across a layer's masks, performance critical. The user-facing concept is a LAYER;
    it maps to exactly one internal object. This is the Lightroom MASKING model (a mask group
    carrying one edit set), NOT a Photoshop pixel-layer stack -- that distinction is what keeps
    it fast (see PERFORMANCE).

    A LAYER = one EditParams (the existing struct, unchanged) + a COLLECTION of mask components
    (radial + brush + linear ...) combined into a single per-pixel alpha + an opacity. The edit
    set applies wherever that combined alpha > 0, so "the same edits across more than one mask"
    falls out for free (the masks are unioned). A layer with NO mask components is GLOBAL
    (applies everywhere) -- so "globally OR on masks" needs no special case.

    NOISE REDUCTION is split across the two stages (the decode pipeline and Develop):
        • Colour NR + a BASELINE luminance NR are GLOBAL-ONLY. They live in the raw decode
          pipeline, applied during demosaic alongside start WB / black / white, so they are
          baked into WorkingImage before Develop runs and are not maskable. (Which engine does
          the decode -- in-house vs macOS Core Image -- is selected by G::decodeRawEngine; both
          converge on WorkingImage so this stays engine-agnostic.)
        • LOCAL luminance NR is a MASKABLE Develop op: an ADDITIONAL luminance-only smoothing
          applied on top of the baseline, per layer/mask. It is a SPATIAL op (below), operating
          on the already-decoded WorkingImage -- so it needs no second decode and is independent
          of which decode engine ran. It is therefore a PORTABLE OpenCV pass (mac + Windows),
          not Core Image. Algorithm: ratio-preserving luma denoise -- Y = dot(rgb, lumaWeights);
          Yd = edge-preserving denoise(Y, strength) (bilateral / guided, NLM as a slower
          higher-quality option); rgb' = rgb * (Yd / Y) so only luminance changes and chroma
          ratios are untouched.

    DATA MODEL (Develop/, planned):
        struct MaskComponent {
            enum Type { Linear, Radial, Brush };
            enum Op   { Add, Subtract };          // union / cut-out WITHIN a layer
            // Linear/Radial: parametric geometry in normalised 0..1 image coords (a few floats)
            // Brush:         a list of stamps {x,y,radius,flow,hardness} -- NOT a bitmap
            bool inverted; bool feather; float featherAmount;
        };
        struct Layer {
            QString name;
            EditParams params;                    // unchanged per-layer unit
            float opacity = 1.0f;  bool visible = true;
            std::vector<MaskComponent> mask;      // empty => global
        };
        struct EditStack {
            EditParams base;                       // global edits (or model as layers[0])
            std::vector<Layer> layers;
            int version = 1;
            bool isIdentity() const;
        };

    COMPOSITING (sequential-on-accumulator). Each layer develops the result of the stack BELOW
    it, blended back by opacity x mask alpha:

        acc = Develop(workingImage, base.params)                 // global foundation
        for each layer i (bottom -> top):
            a   = opacity_i * maskAlpha_i(x)                     // 0 where the layer doesn't act
            acc = lerp(acc, Develop(acc, params_i), a)           // per pixel

    This makes edits sequentially ADDITIVE (each layer sees the result below it) while keeping
    opacity and masks well defined per pixel. Layer ORDER therefore matters and REORDER changes
    the result -- that is correct and is surfaced in the UI. Base layer is always opacity 1, no
    mask.

    PERFORMANCE (the reason for the masking model over a pixel-layer stack). Ops are split by
    cost and the cheap ones are FUSED:
        • POINT ops -- exposure, contrast, highlights/shadows/whites/blacks, temp, tint, the
          Colour RGB + HSL sliders, and (for now) texture + dehaze -- are pure per-pixel
          functions. ALL layers' point ops run
          in a SINGLE fused pass over the image: one read, one write, no per-layer intermediate
          buffers; N layers cost N cheap evaluations in registers per pixel (bandwidth-bound).
          This is the slider-drag hot path and is independent of layer/mask count.
          (DECISION 2026-06-25: texture + dehaze are POINT ops for now to keep the fused engine
          simple; revisit a SPATIAL local-contrast version later -- see RawDecode_ToDo #15.)
        • SPATIAL ops -- local luminance NR (and a future spatial texture/dehaze) -- need a
          neighbourhood, so they cannot fuse: one pass each, blended by the layer's mask, CACHED
          and recomputed only when their own param or mask changes (never on a point-slider
          drag). The pass output depends only on its strength param, not the mask VALUES, so it
          is computed over the mask BOUNDING BOX + filter-radius halo (not the whole frame) and
          the cached result is reused while only the blend re-reads the mask alpha.

    THREADING. The fused point pass is data-parallel WITHIN one render() (row-tiles), NOT
    spread across the ImageDecoder worker pool. The decoder pool is task-parallel ACROSS images
    (one decoder per image, to fill the browse cache); an interactive edit is a SINGLE image, so
    the pool leaves it on one core -- only splitting that one image's rows uses all cores.
    IMPLEMENTED (phase 1, Develop::applyPointOps) with QtConcurrent::run over the global
    QThreadPool -- the same idiom FocusStack already uses (fs.cpp) -- rather than
    cv::parallel_for_ (which appears nowhere in the tree). The global pool is fixed-size, so a
    background decoder that also calls render() shares it and oversubscription is bounded
    automatically (no N x N blow-up, no setNumThreads juggling). Rows split into <= maxThreads
    chunks of >= 64 rows; small images run flat. (cv::parallel_for_ remains an option if a later
    spatial op is implemented in OpenCV and wants to stay on OpenCV's own pool.)

    MASKS store GEOMETRY, never bitmaps. Parametric masks (linear/radial, ~6 floats) are
    evaluated analytically inline in the fused loop -- zero storage, resolution-independent.
    Brush masks are stored as the STROKE/STAMP LIST the user painted (not a rasterised alpha),
    rasterised to a cached alpha buffer on load and after each edit: tiny sidecar, resolution-
    independent (rasterise at proxy vs full res), and serialisable as text in the XMP sidecar. A
    layer's combined alpha (the add/subtract union of its components) is rasterised ONCE and
    cached, recomputed only when mask geometry changes -- never when sliders move.

    CACHE TIERS (most already present):
        1. pre-develop WorkingImage          -- Develop/workingimagecache.h (DONE)
        2. per-layer combined mask alpha      -- recompute on geometry change only
        3. spatial-op outputs (local luma NR ...) -- recompute on their param/mask change only
        4. fused point pass                   -- always re-run on a point-slider drag (cheap)
    Plus a downscaled PROXY developed during an active drag, full-res async on release (now
    implemented: renderDevelopFullResAsync, see "Load Responsiveness") -- sub-frame with the
    fused engine.

    PERSISTENCE -- XMP SIDECAR, per image. Winnow already writes per-image .xmp sidecars
    (Metadata/xmp.cpp Xmp::writeSidecar, DOM-based); the EditStack serialises there (parametric
    geometry + brush strokes as numbers, Lightroom-style). This REPLACES the current global
    QSettings "Develop/Layers/<name>/..." persistence (which is per-machine, not per-image -- the
    scaffold's main correctness gap). EditStack is versioned for forward migration.

           one EditParams) but Develop now does a single parallel per-pixel pass. Wired live:

           blue, tint lowers green toward magenta; sliders -100..100 normalised to +/-1, full
           scale = +/-50% on a channel via kTempGain/kTintGain). WB and exposure are both linear
           multiplies, so buildPointCoeffs folds them into one per-channel gain (PointCoeffs::
           channelGain) applied before the perceptual tone curve. Exposure = 2^EV scene-linear
           gain.

           TONE CURVE (contrast + highlights/shadows/whites/blacks) -- all four tone regions plus
           contrast are pure 1-D functions of a channel's white-normalised value, applied in a
           PERCEPTUAL (~gamma 2.2) domain, NOT scene-linear (a linear-space curve about 0.18
           explodes highlights, which sit several stops above the pivot; encoding to gamma first
           puts mid-grey near 0.46 so tones shape evenly). buildPointCoeffs BAKES the whole chain
           into a per-render LUT (PointCoeffs::toneLut, indexed by s=(v/white)^(1/gamma), returning
           white-normalised linear), so the hot loop does ONE encode pow + one interpolated lookup
           per channel instead of several pow/exp -- the tone pass is now cheaper than the old
           contrast-only path. Contrast = gentle slope (1 + 0.4*amount, [0.6,1.4]) about mid-grey;
           the -100..100 slider is NORMALISED to amount -1..1 (without this the first slider unit
           already hits the extreme). Each tone region adds a Gaussian-weighted lift in perceptual
           space (centres: blacks 0.0, shadows 0.25, highlights 0.75, whites 1.0; kToneSigma 0.18,
           max lift kToneShift 0.20 at full slider; + brightens / - recovers, Lightroom-like).
           The LUT spans up to ~3 stops above white (kToneLutMaxN) so highlight headroom is shaped,
           not clipped.

           COLOUR (RGB + HSL, under the Color header) -- all six are POINT ops, so they ride the
           same fused pass at ~no extra cost. RGB (red/green/blue, -100..100) are per-channel
           scene-linear gains folded straight into channelGain alongside WB + exposure (kRgbGain
           0.5 -> full slider = +/-50% on a channel). HSL (hue/saturation/luminance, -100..100) is
           a cross-channel block applied AFTER the tone curve (it mixes channels, so unlike the
           tone curve it cannot be a per-channel LUT): hue = a Rodrigues rotation about the neutral
           (1,1,1) axis so greys stay grey (kHueMaxRad = 60deg at full); saturation = chroma scale
           about Rec.709 luma (-100 -> grey, +100 -> 2x); luminance = uniform gain (-100 -> black,
           +100 -> 2x). hslActive gates the block; identity sliders leave the pass untouched.

           SPATIAL ops (Texture #6, Dehaze #7) run AFTER the fused point pass, each owning a
           full-image pass (like Denoise) -- they need a neighbourhood so cannot join the per-
           pixel kernel. Both are ratio-preserving on LUMINANCE (chroma untouched) and scale their
           blur radius to the image's longer edge so the screen-res proxy and the full-res settle
           render shape the same frequency band. Texture = mid-frequency local contrast: a Gaussian
           base (kTextureSigmaFrac ~0.0015*edge) splits a detail band that + amplifies (kTextureGain)
           and - smooths. Dehaze = APPROXIMATION (not dark-channel-prior; user-chosen): a large-
           radius local-contrast boost (box blur, kDehazeSigmaFrac ~0.02*edge -- box because its
           running-sum cost is ~radius-independent) + a contrast pull about a low pivot (kDehazePivot
           0.3, deepens shadows) + a saturation lift (kDehazeSat); + clears haze, - adds it. These
           are global (no-mask) now; the layer compositor will bound them to a mask bbox later.
           Costs land only on the full-res settle render (the proxy filters ~3MP); denoise

               DevelopProperties::setActiveMaskParams (stores into the component, debounce-persists, no
               tree rebuild). Feather/Invert live-update via maskFeatherChanged/maskInvertChanged; tint
               colour conveys op (Add=red, Subtract=blue); Invert flips (Radial: effect outside).
             - RENDER COMPOSITING (Stage 4b + N-layer stack): the render composites the WHOLE stack
               in SCENE-LINEAR before the output transform, INDEPENDENT of which layer is active for
               editing -- so a saved mask is visible the moment the image loads (no need to select
               its layer). DevelopProperties::stackJob() captures it as plain values (Base params +
               each enabled non-Base layer's params/masks/combine) so it is safe for the off-thread
               full-res render. WorkingImageCache::renderStack starts from Base (applied globally),
               then for each layer develops `work` with that layer's params (an identity side aliases
               work -- no copy/Develop) and blends over the accumulator by its mask
               (acc = acc*(1-m)+layer*m; an empty/global mask replaces below); one final
               OutputTransform. Each layer develops the ORIGINAL work (independent, not stacked
               adjustments), so where a layer's mask=1 it replaces what is below. buildMaskBuffer
               (mainwindow.cpp) rasterizes a layer's component(s) at WorkingImage (pre-orientation)
               resolution; each pixel maps work-normalized -> output-normalized (output = work
               rotated CW by degrees) so geometry edited on the oriented loupe lines up after the
               EXIF rotation. parseMaskComp/evalMaskComp branch by tool: Linear = projection ramp;
               Radial = distance in the ellipse's local frame (computed in OUTPUT-PIXEL space so the
               ellipse keeps aspect + rotation), 1 inside -> 0 at the boundary. The BRUSH component is
               not analytic: its strokes are RASTERIZED (Develop/brushstamp.h, shared with the
               ImageView preview so they match) into a work-space coverage buffer, cached
               (brushRasterCached, paramsJson+dims+degrees key, mutex-guarded since the GUI proxy and
               full-res worker both build it; only proxy-size buffers are cached -- full-res is ~w*h*4
               bytes and built once per settle). Components combine in order: brush samples its buffer
               (then component Invert), parametric uses evalMaskComp. Falloff smootherstep,
               Add=union(max), Subtract=*(1-c), Invert flips. developComposite remains the fast
               single-pass path when there are no non-Base layers. Mask add/delete/geometry/feather/
               invert and a committed brush stroke emit paramsChanged() to re-render. Deferred: true
               STACKED adjustments (each layer builds on the result below, not the original), the
               combine field (Union only today), and AI tools.
             - BRUSH (freehand mask): vector strokes in paramsJson {size,flow,autoMask,strokes:[{pts,
               size,feather,flow,erase,autoMask}]} (each stroke snapshots its settings; current
               settings drive the next stroke). Painted on the loupe (ImageView): a size-circle cursor,
               left-drag paints, Alt-drag erases; dabs = smootherstep feathered circles accumulated per
               stroke by MAX then composited with flow (build-up across strokes). Live red/blue tint
               preview rebuilt only on the dirty segment bbox; the develop re-composites on stroke
               release (maskGeometryChanged -> setActiveMaskParams). Dock rows: Size/Feather/Flow/Auto
               mask/Invert. Size also via [ / ] keys or a two-finger drag (ImageView wheelEvent, gated
               to brush+hover) -> maskBrushSizeRequested -> setActiveBrushSize syncs the dock; "A"
               toggles auto-mask (-> maskBrushAutoMaskRequested -> setActiveBrushAutoMask); Cmd/Ctrl+Z
               removes the last stroke. ImageView needs focus (set on brush enter) and claims [ ] A
               from their global action shortcuts via QEvent::ShortcutOverride. Cursor hint: dashed
               green = auto-mask, pink = erase. AUTO-MASK (WIP -- luminance-only, fixed tolerance,
               coarse guide; usable but expected to be refined): a luminance guide (BrushStamp::Guide, ~1024
               long edge, output-oriented) is built once from the loupe pixmap when the brush
               activates (ImageView::ensureAutoGuide) and registered by path (BrushStamp::putGuide);
               the develop render samples the SAME guide (getGuide(fPath), threaded through
               developCompositeStack/buildMaskBuffer/brushRasterCached) so preview == render. A
               flagged stroke attenuates each dab by an edge factor: paint only where |lum - lumRef|
               < tol (lumRef = guide luminance at the stroke start; smootherstep falloff). Render
               compositing as above.
             - LIFECYCLE / FLUSH: MW::fileSelectionChange calls setCurrentImage; after the loupe
               pixmap is shown (both the cache-hit path and MW::refreshViewsOnCacheChange) MW::
               applyDevelopPreviewIfEdited renders the saved edits over an edited image (no-op for
               identity, so an unedited image stays the normal decode -- and is never written).
               Sidecars are flushed on navigate-away (setCurrentImage), quit (closeEvent), and
               before ingest/copy (MW::ingest / copyFiles); NEVER during a drag. Writes are
               synchronous (a few-KB text file). The OPTIONAL debounce-while-editing write (~2 s
               after edits settle) is gated by G::isDevelopDebounceWrite (default true); the other
               flushes always run.
             - PERF NOTE: setCurrentImage reads the sidecar synchronously on first touch of each
               image (cheap: no sidecar => fast empty read). Folding the develop blob into the
               existing metadata sidecar read (a datamodel column) is a fast-follow if navigation
               over many edited images shows latency.
           Increment 2 (presets): snapshot an EditStack (or subset) to a QSettings presets
           collection; apply to a selection (write each image's sidecar). Then masks/compositor.
        5. Brush masks (stroke list + rasterise) -> spatial ops with cache tier 3 -> proxy-on-drag.

------------------------------------------------------------------------------------------
Develop dock (UI)
------------------------------------------------------------------------------------------

    The Develop panel is a dock (View menu > Develop Editor Panel, shortcut F9), built like
    the Embellish dock: MW::createDevelopDock() in initialize.cpp wraps a DevelopProperties
    widget (Develop/Properties/), a PropertyEditor subclass mirroring EmbelProperties. It is
    tabbed with the Embellish dock in the right dock area and its visible / collapsed state
    persists exactly as the other docks (settings isDevelopDockVisible + DevelopDock, plus the
    workspace struct field isDevelopDockVisible).

    The dock's widget is a container, top to bottom: a fixed-height SCOPES strip, the TRANSFORM
    (crop + perspective) panel, then the DevelopProperties tree taking the remaining (stretch)
    space. The scopes and transform strips are each independently toggled from the editor bar.

Develop scopes (histogram + vectorscope)
------------------------------------------------------------------------------------------

    A live histogram (left) and vectorscope (right) sit above the property tree, in
    Develop/Scopes/: HistogramView (256-bin R/G/B with additive blend + a faint luma outline),
    VectorscopeView (YCbCr Cb/Cr 2-D accumulation on a circular graticule, log-scaled and
    hue-coloured), wrapped by ScopesView (a QHBoxLayout, histogram given the larger stretch).

    They reflect the image CURRENTLY SHOWN in the loupe. MW::updateDevelopScopes(const QImage&)
    does ONE strided sample pass (~180k-pixel budget, so a 50MP RAW costs the same as the proxy;
    constScanLine so no detach) that fills both scopes from the same downsample. It is called
    after each develop preview render (the post-Develop `out`, both setDevelopPreview sites) and,
    for an unedited image, from the decoded image in applyDevelopPreviewIfEdited; the scopes are
    cleared at the imageView->clear() sites. No new timers -- it piggybacks the existing coalesced
    render path. The pass is a cheap no-op while the scopes are hidden.

    A button on the Develop editor bar (graphic.png) toggles the strip via MW::toggleDevelopScopes;
    the choice persists as Settings "Develop/scopesVisible". (A standalone widget above the tree was
    chosen over an in-tree spanned row so the scope stays pinned rather than scrolling away; see
    notes/Develop_Scopes_Plan.txt for the rejected alternative.)

    Tone-region slider (ToneRegionSlider, under the histogram, sharing its x-axis): a black->white
    track with three draggable handles that split the tonal range into blacks | shadows |
    highlights | whites (like the Lightroom tone-curve region splits). The handles move WHERE the
    shadows/highlights tone controls act and how far each reaches; blacks (0) and whites (1) stay
    pinned. They map to three EditParams fields (toneShadowCenter / toneCrossover /
    toneHighlightCenter, perceptual 0..1, defaults 0.25/0.50/0.75). In Develop::buildToneLut the
    shadows/highlights Gaussian centres come from the two outer handles and each region's sigma is
    derived from its gap to the crossover (kToneSigmaSpan, so the default gap reproduces the old
    fixed kToneSigma) -- i.e. the defaults are an exact no-op. A drag writes the active layer's
    params and emits paramsChanged() (live re-render, coalesced like the sliders); the splits only
    change pixels when a tone slider is non-zero, so they are NOT part of EditParams::isIdentity().
    DevelopProperties::bindToneSlider() connects the slider (drag -> onToneSplitsChanged) and
    populateSlidersFromStack() pushes the saved positions back on image switch.

    Cursor readout: hovering the loupe marks the pixel under the cursor on both scopes -- vertical
    R/G/B ticks on the histogram, a ring at its Cb/Cr on the vectorscope. ImageView (mouse tracking
    on) emits cursorImagePos (normalized) / cursorLeftImage; MW::onImageCursorPos samples that one
    pixel from developShownImage (the cached QImage last handed to updateDevelopScopes == what the
    loupe shows; implicitly shared, so O(1) per move, no re-bin, no pixmap->image copy) and calls
    ScopesView::setMarker / clearMarker. Cleared when the cursor leaves the image. No-op while the
    scopes are hidden.

    Vectorscope zoom: a right-click menu (100/150/200%) magnifies the plotted trace + marker
    outward from the centre to reveal low-saturation detail; the graticule circle stays as the
    100% reference and a small "%"-label shows when zoomed. VectorscopeView::setZoom +
    zoomChanged; ScopesView re-emits as vectorscopeZoomChanged; MW restores/persists it as
    Settings "Develop/vectorscopeZoom".

    Skin-tone line: the same right-click menu toggles a flesh-tone reference line -- the YIQ
    I-axis, which from our BT.601 Cb/Cr coefficients is (Cr-128) = -1.946*(Cb-128); skin of every
    ethnicity falls along it (hue is constant; only saturation/brightness vary). Drawn centre ->
    edge into the warm upper-left ray, independent of zoom (it is an angle guide). setSkinLine +
    skinLineChanged -> ScopesView::vectorscopeSkinLineChanged -> Settings
    "Develop/vectorscopeSkinLine".

Develop Transform panel (crop + perspective)  [UI PROTOTYPE]
------------------------------------------------------------------------------------------

    TransformPanel (Develop/Transform/) is a compact control strip below the scopes and above
    the property tree -- the sibling of ScopesView. It is the UI surface for a non-destructive
    crop/straighten/perspective edit; it carries NO geometry or pixel state. The crop rectangle,
    straighten angle and perspective homography will live in the edit stack and apply LAST in the
    pipeline (after the develop ops), so they stay non-destructive and masks map through them. The
    image-side overlay and the warp engine are wired to the panel's signals in a later increment;
    for now each control persists its own state and emits a signal (the integration hook).

    Controls: an Aspect combo (As shot / 1x1 / 3x2 / 4x3 / 5x7 / 4x5 / 8.5x11 / 16x9 / 16x10 /
    persisted custom aspects / "Add custom aspect..."); userData per item is the w/h ratio
    (0.0 = As shot/free). A padlock toggle locks the ratio. A "Level" tool (draw a line to
    straighten), a checkable "Crop" tool (drag a rectangle; Alt/Option-drag gives a 4-point
    polygon), and a "Rectify" button (warp the whole image so the polygon is axis-aligned, then
    fit the largest inscribed rectangle as the new crop -- user-confirmed behaviour). A "Fill
    extra canvas" checkbox (content-aware edge-extend of empty corners left by straighten/rectify).
    A [?] tips button (top-right, "Crop and transform tips").

    Shortcuts: "R" toggles the panel (MW::toggleDevelopTransform, developTransformAction, editor-
    bar button rectangle.png; persists Settings "Develop/transformVisible"). "A" toggles the
    aspect lock while the panel (or one of its non-text controls) has focus -- handled by
    TransformPanel::eventFilter accepting the key's ShortcutOverride, NOT a QAction, because a
    panel-scoped "A" QAction would be an ambiguous overload with the window-level "A" (Run
    Droplet) and the brush "Auto mask A". Custom aspects + lock + fill-canvas state persist under
    Settings "Develop/Transform/".

    Crop rubber-band tool (ImageView) -- IMPLEMENTED:
    Showing the Transform panel (R / button) calls ImageView::beginCropEdit; hiding it calls
    endCropEdit. KEY RULE (user-specified): the crop tool NEVER changes the view transform -- no
    zoom, no auto-pan. The user keeps full control of zoom/pan; begin/end only toggle the overlay
    and leave the image exactly where it is. cropN (normalized image coords) is the source of truth;
    cropFrameVp (the on-screen frame, viewport px) is DERIVED from it (cropNToVpRect) and tracks the
    image under the current zoom/pan. Interaction (all guarded by cropActive()):
        - drag a HANDLE (cropHitTest 0..7: 4 corners + 4 edge mids) -> resizes the frame over the
          STATIC canvas (cropResizeFromHandle anchors the opposite corner/edge, clamps to the
          image's on-screen rect); cropN := cropVpRectToN(frame). No re-stage, no zoom change.
        - drag elsewhere -> a NORMAL canvas pan (the existing scrollbar pan); the frame stays fixed
          on screen and the image slides under it, so cropN is recomputed from the unchanged frame.
          This is how the crop is repositioned -- you move the canvas, not the crop rectangle.
        - aspect lock (from the panel, setCropAspect) constrains the frame ratio; "As shot"/ratio 0
          is free.
        - when the USER zooms (scale()) or the window resizes, the frame is re-derived from cropN
          (cropSyncFrameFromN) so the crop stays on the same image content.
    Overlay (cropDrawOverlay, viewport coords in drawForeground): dimmed exterior, rule-of-thirds
    grid, frame border, corner/edge handles. ImageView emits cropChanged(x,y,w,h) for persistence.
    The wheel is swallowed during crop (wheel = image navigation, which would strand the overlay).

    Warp (4-point perspective) sub-mode -- IMPLEMENTED (interaction only):
    Holding Alt/Option while dragging a crop CORNER breaks the rectangle into a free quadrilateral
    (cropWarp=true; cropEnterWarp seeds cropQuadN/cropQuadVp from the rectangle, corners TL,TR,BR,BL).
    In warp mode only the 4 corners are handles (each drags independently, clamped to the image);
    cropHitTest / cropResizeFromHandle / cropDrawOverlay / cropSyncFrameFromN all branch on cropWarp.
    The overlay draws the quad with a bilinear 3x3 guide grid + 4 corner handles and dims outside the
    polygon. Panning still works (the quad is fixed on screen, recomputed from the moving image; the
    pan-clamp uses the quad's bounding box, cropFrameBBoxVp). Choosing an aspect, re-entering, or
    Rectify leaves warp mode.

    Perspective-warp PIXEL engine -- IMPLEMENTED (Develop/Transform/croptransform.{h,cpp}):
    CropTransform::rectifyPerspective(src, quadPx[4], &outCropNorm) warps the whole image so the
    quad (TL,TR,BR,BL image-pixel coords) becomes an axis-aligned rectangle and returns the
    corrected QImage (transparent where the warp leaves the original) plus the largest inscribed
    axis-aligned rectangle (normalized) as the default crop. Math: cv::getPerspectiveTransform
    (quad -> WxH rect sized by averaged opposite-edge lengths) -> translate into a positive canvas
    bounded by the warped image corners -> cv::warpPerspective (BGRA, transparent border) ->
    validity mask (alpha>=250) -> largest-rectangle-in-binary-matrix (histogram+stack, on a
    downscaled mask). Self-contained (OpenCV only) so it can later be called from the render
    pipeline. Validated offline vs the same algorithm in cv2 (keystone quad -> exact axis-aligned
    rect; inscribed rect avoids the transparent corners). ImageView::rectifyCrop drives it on the
    DISPLAYED image: warp -> setPixmap -> resetFitZoom -> re-enlarge scene -> crop continues on the
    rectified image (cropN = outCropNorm). This is a DISPLAY-LEVEL preview (the develop pipeline
    still holds the original; a develop edit re-renders un-warped).

    Persistent crop (geometry applied in the render pipeline) -- IMPLEMENTED (Phase A, rectangle
    crop):  EditStack now carries a per-image Geometry (cropX/Y/W/H, straighten, hasWarp+quad[8],
    fillCanvas), serialized in the sidecar (toJson/fromJson, folded into isIdentity). The renderer
    applies it LAST -- developCompositeStack runs CropTransform::applyGeometry(out, job.geometry)
    after the develop ops + EXIF orientation, on the full-output-dimension image, in both the proxy
    and async full-res paths (StackRenderJob.geometry from DevelopProperties::stackJob).
    Commit model (Lightroom): MW::developCropEditing gates it -- while the crop tool is being edited
    the render NEUTRALIZES the geometry so the FULL frame shows for the overlay; committing applies
    it. MW::enterDevelopCrop (set flag -> render full -> ImageView::beginCropEdit(storedCrop)) and
    MW::exitDevelopCrop (read cropRect -> endCropEdit -> DevelopProperties::setCurrentGeometry ->
    clear flag -> render cropped) are driven from toggleDevelopTransform + developDockVisibilityChange.
    ImageView::setDevelopPreview re-fits when the image dimensions change (crop applied/removed); the
    same-size path (a develop-slider drag) still just swaps pixels.

    Persistent warp (Phase B) -- IMPLEMENTED: the 4-point perspective is a STORED geometry param
    (hasWarp + quad[8]), non-destructive (no pixmap bake). Two-phase display via the render gate:
    while the crop tool is active the render suppresses ONLY the crop (resets it to full) but KEEPS
    the warp, so before Rectify the full frame shows (trace the quad) and after Rectify the corrected
    canvas shows (set the final crop). MW::rectifyDevelopCrop stores g.hasWarp + g.quad (= the traced
    corners, normalized in the developed+oriented frame = applyGeometry's input space) + g.crop (the
    suggested largest-inscribed rect from ImageView::computeRectifyCrop), then re-renders the
    corrected canvas and restarts the crop overlay on it. exitDevelopCrop keeps the stored warp and
    updates only the crop. KNOWN LIMIT: re-warping an already-warped image is wrong (the new quad is
    traced on the warped canvas, but applyGeometry warps the un-warped frame) -- there is no
    warp-reset UI yet, so fresh-image warp is the supported path.

    NOT yet done (next increments): the straighten/Level tool; a warp-reset (to re-warp); confirm
    export applies the geometry; content-aware fill. The panel's Crop and Level buttons are unwired.

    Sections (built via PropertyEditor::addItem):
        Layers   header with -/+ buttons (same idiom as Embellish Templates) + a "Select
                 layer" combo. A layer is one set of adjustments; switching rebuilds the
                 Basic/Effects rows from that layer's saved values. Layers persist under
                 Settings "Develop/Layers/<name>/..." with "Develop/Layers/current".
                 SCAFFOLD ONLY -- this is the "layers as ALTERNATIVES" model (only the selected
                 layer is active, persisted per-machine in QSettings). It is superseded by the
                 "Layer & masking model (planned)" subsection above: all layers active as a
                 composited stack, the combo selects only which layer the sliders EDIT, opacity +
                 masks per layer, persisted per-image in the XMP sidecar.
        Basic    Denoise (checkbox); Temp, Tint, Exposure, Contrast, Highlights, Shadows,
                 Whites, Blacks, Texture, Dehaze (sliders). Integer -100..100 except Exposure
                 (EV, -5.00..5.00). All default to 0 (identity).
        Color    Red, Green, Blue (per-channel gain); Hue, Saturation, Luminance (global HSL).
                 Integer -100..100 sliders, all default 0 (identity).
        Effects  header only (placeholder for future creative effects).

    Each layer maps onto one EditParams (DevelopProperties::editParams()). Binding a value
    change to a re-decode is deferred: itemChange() persists the value to settings and emits
    paramsChanged() so the hook is ready.
