INITIALIZATION

    Persistent settings are saved between sessions using QSettings. Persistent settings
    form two categories at runtime: action settings and preference settings. Action
    settings are boolean and maintained by the action item's setChecked value.
    Preferences can be anything and are maintained as public variables, mostly in MW
    (MainWindow) and managed in the PrefDlg class.

    Not all settings can be read immediately, as the classes that use them may not yet
    have been created. Examples are IconView parameters and the recent folders menu
    list, since IconView and the menus have not been created yet. Therefore some
    settings are assigned up front and the rest are assigned as needed.

------------------------------------------------------------------------------------------
Startup Sequence
------------------------------------------------------------------------------------------

    • Load QSettings
    • Set default values if no settings available (first time run)
    • Set preference settings including:
        • General (previous folder etc.)
        • Slideshow
        • Cache
        • Full screen docks visible
        • List of external apps
        • Bookmarks
        • Recent folders
        • Ingest history folders
        • Workspaces
    • Create actions and set checked based on persistent values from QSettings
    • Create bookmarks with persistent values from QSettings
    • Update external apps with persistent values from QSettings
    • Load shortcuts (based on being able to edit shortcuts)
    • Execute updateState function to implement all persistent state settings

    • Select a folder
        • Load datamodel with QDir info on each image file
        • Add the rest of the metadata to datamodel (incl. thumbs as icons)
        • Update the image cache

FOLDER LOADING PIPELINE

    Use G::isFlowLogger or G::isLogger to trace program progress.

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    Loading a folder runs in two phases that do NOT overlap with each other.
    Phase 1 populates the DataModel with one row per eligible file and runs on
    the GUI thread. Phase 2 reads metadata and icons (MetaRead, on worker
    threads) and caches full-size images (ImageCache, on worker threads); these
    two activities within Phase 2 DO overlap.

    Phase 1 - Build the DataModel (GUI thread)

    • A folder or bookmark click calls MW::folderSelectionChange. The mouse
      modifier decides whether the folder replaces the current selection
      (resetDataModel = true) or is added to it.

    • When resetDataModel is true the prior load is torn down first: MW::stop
      halts the worker threads, MW::reset clears state, and
      DataModel::clearDataModel empties the model. clearDataModel calls
      QCoreApplication::removePostedEvents(this, QEvent::MetaCall) BEFORE
      clear(). QStandardItemModel coalesces item edits into a posted
      _q_emitItemChanged meta-call; if that (or a stale worker setData/setIcon
      meta-call) fired after clear() freed the items it would dereference freed
      pointers and crash. DataModel::newInstance then bumps dm->instance so any
      surviving stale worker signals are ignored.

    • Subfolders are counted (Utilities::subFolderTree) and the folder plus,
      in the recursive case, every subfolder are queued by
      DataModel::enqueueFolderSelection / enqueueOp. macOS .photoslibrary
      bundles are filtered out (they hold huge numbers of derivative files).

    • DataModel::scheduleProcessing starts DataModel::processNextBatch, which
      drains the folder queue in time-sliced batches. It yields to the event
      loop every ~30 ms (kTimeBudgetMs), with a secondary cap of 64 folders per
      tick, rescheduling itself with a zero-delay timer. This time-slicing is
      what keeps the GUI responsive while a large recursive tree loads.

    • DataModel::addFolder handles one directory: dir.entryInfoList() lists
      the eligible files, they are sorted (DataModel::lessThan, or
      lessThanCombineRawJpg when raw+jpg combining is on), and each file is
      appended as a row (setRowCount + DataModel::addFileDataForRow, which fills
      the OS file columns and performs raw+jpg pairing).

      Cost note: the sort/filter proxy (dm->sf) inserts each new row into sorted
      position at O(N), so the whole load is O(N^2). Time-slicing bounds each
      GUI freeze to roughly one folder's insert time but does not change that
      total; a single very large folder loaded late can still block briefly.

    • When the queue empties DataModel emits folderChange, handled by
      MW::folderChanged. It re-enables the panels, sets the current index, and
      either finishes immediately (DataModel::isAllMetadataAttempted, e.g. a
      pure removal with no new rows) or starts Phase 2.

    Phase 2 - Read metadata/icons and cache images (worker threads)

    • MW::folderChanged starts MetaRead with a queued
      QMetaObject::invokeMethod(metaRead, "initialize") followed by
      MetaRead::setStartRow(startRow). MetaRead lives on metaReadThread.

    • MetaRead::dispatchReaders / dispatch assign rows to Reader workers,
      walking outward from startRow (the current image) so the user's position
      is served first. Each Reader reads metadata (Reader::readMetadata ->
      DataModel::addMetadataForItem) and a G::maxIconSize (256 px) icon
      (Reader::readIcon -> DataModel::setIcon1, stored in Qt::DecorationRole),
      delivered to the GUI thread over queued connections. Backpressure
      (dm->queuedReaderEvents, cap 4 x readerCount) defers dispatch when the GUI
      thread falls behind draining those events.

    • Once the start row itself has been read, MetaRead emits select, which
      routes Selection::setCurrentIndex -> MW::updateChange ->
      MW::fileSelectionChange. That loads the current image and starts
      ImageCache, so ImageCache runs concurrently with the rest of MetaRead.

    • ImageCache caches full-size images ahead of and behind the current
      position: ImageCache::setCurrentPosition -> updateTargets ->
      launchDecoders -> decodeNextImage -> ImageDecoder (decode / rotate /
      colorManage) -> ImageCache::fillCache -> cacheImage -> nextToCache.

    • MetaRead::done signals MW::folderChangeCompleted, which builds the
      filters, re-enables the sort/filter menus, applies a single sort pass, and
      resizes the table columns.

    Instances and aborting

    • Every new primary folder increments dm->instance
      (DataModel::newInstance). Worker signals carry the instance they were
      issued under; DataModel slots (setIcon1, addMetadataForItem, setValSf,
      ...) drop any call whose instance no longer matches, so lagging signals
      from an abandoned load are harmless.

    • G::stop and dm->abort are the cooperative stop flags. MW::stop sets
      them, signals the workers to abort, and waits (event-driven, 3 s cap)
      for them to acknowledge before resetting.

------------------------------------------------------------------------------------------
Concurrent Flow - New Folder Selected
------------------------------------------------------------------------------------------

    • Selecting a new folder in the Folders or Bookmarks panel calls
      MW::folderSelectionChange.

    • If the folder replaces the current selection, the prior load is torn
      down: all worker threads are stopped, the DataModel is cleared and all
      parameters are reset (MW::stop, MW::reset, DataModel::clearDataModel).

    • The DataModel is loaded from the folder queue by
      DataModel::processNextBatch, which calls DataModel::addFolder and
      iterates DataModel::addFileDataForRow to populate basic file info per
      row. processNextBatch is time-sliced (yields every ~30 ms) so the GUI
      stays responsive. MW::folderChanged then sets the current index.

    • MetaRead::setStartRow / dispatchReaders / dispatch send Reader workers
      outward from the start row, loading metadata
      (-> DataModel::addMetadataForItem) and 256 px icons
      (-> DataModel::setIcon1).

    • When the start row has been read, MetaRead emits select, which (via
      MW::updateChange -> MW::fileSelectionChange) starts ImageCache. From
      there ImageCache walks setCurrentPosition -> updateTargets ->
      launchDecoders -> fillCache, caching concurrently with MetaRead.

------------------------------------------------------------------------------------------
Concurrent Flow - New Image Selected
------------------------------------------------------------------------------------------

    A new image is selected, which triggers fileSelectionChange:

    • If starting the program, select the first image in the folder.
    • Record the current datamodel row and its file path.
    • Update the thumbView and gridView delegates.
    • Synchronize the thumb, grid and table views. If in loupe mode, load
      the current image.
    • Update window title, statusbar, info panel and classification badges.
    • Update the metadata and image caching.
    • If the metadata has not been cached yet for the selected image
      (usually the first in a new folder), load the thumbnail.
    • Update the cursor position on the image caching progress bar.

    A new image is selected which triggers a scroll event:

    • IconView selection change may trigger a scroll signal, based on the
      current view (thumb, grid or table). MW::thumbHasScrolled /
      gridHasScrolled / tableHasScrolled are signalled.
    • Update the icon range (firstVisible / lastVisible).

------------------------------------------------------------------------------------------
Icon Chunk Size
------------------------------------------------------------------------------------------

    Icons or thumbnails, 256 px on the long side, are stored in the DataModel
    in column 0 under Qt::DecorationRole (the QPixmap); the icon's QRect is
    kept on the same cell under G::IconRectRole. The number of icons to cache is
    dm->iconChunkSize, set per folder in DataModel::resolveIconChunkSize: in brute
    force (G::useJitIconCache == false) it equals the row count (all icons); in JIT
    it is the memory budget (iconBudgetCount), clamped to the row count and to the
    3x-visible floor (iconChunkFloor). See the "Icon Caching" section below for the
    full brute-force vs JIT model (Layers 1-3). As the user moves through the images,
    either by selection or scrolling, the icon range is recalculated based on the
    midVisibleIcon, and iconChunkSize is held at the floor by MW::updateIconRange.

    When the user progresses through the images, the current image is centred
    in the visible views (thumbView, gridView and tableView) using the
    midVisibleCell.

------------------------------------------------------------------------------------------
Tracking Icon Parameters
------------------------------------------------------------------------------------------

    The number of visible icons (visibleCellCount) can change whenever:

        • the size of the view changes
        • the size of the icons changes

    When this happens:

        • IconView::updateVisible is called. A number of IconView operations
          use the parameters defined in IconView::updateVisible.
        • MW::updateIconRange updates the DataModel iconChunkSize based on
          which view has the most visible icons and the current iconChunkSize.

    The first, mid and last visible icons change whenever there is a scroll
    event. The scroll event can be triggered by the user scrolling or
    selecting another image. See also SCROLLING below.

        • MW::thumbHasScrolled, MW::gridHasScrolled or MW::tableHasScrolled is
          triggered.
        • MW::updateIconRange calls IconView::updateVisible.
        • Other visible views are scrolled to sync.

CACHING

    Metadata caching has been replaced by `MetaRead` (`Cache/metaread.cpp`) dispatching a
    pool of `Reader` workers (`Cache/reader.cpp`). The earlier `MetadataCache` /
    `metadataCacheThread` system (`Cache/mdcache.cpp`, `Cache/mdcacher.cpp`,
    `Cache/mdcachemgr.cpp`) is no longer compiled — those files are not listed in
    `winnow64.pro` and are retained on disk for reference only.

------------------------------------------------------------------------------------------
Overview
------------------------------------------------------------------------------------------

    • When a new folder or a new file is selected, or when a scroll event occurs in
      IconView, MetaRead is started. MetaRead dispatches Reader workers to load
      metadata and icons into the DataModel.

    • As metadata accumulates, MetaRead signals ImageCache::addCacheItemImageMetadata
      so the image cache can begin loading full-size images. When MetaRead crosses
      imageCacheTriggerCount it signals the ImageCache to start.

    • If a new image is selected, MW::fileSelectionChange repositions MetaRead and the
      ImageCache around the new current row.

    • If the user scrolls in the thumbView or gridView, scroll events are sent to
      MW::thumbHasScrolled / MW::gridHasScrolled. A delay ensures that rapid scrolling
      coalesces — the next scroll event supersedes the prior, and MetaRead is only
      re-dispatched when there is a pause or the scrolling is slow. ImageCache is not
      restarted here because a new image has not been selected.

    See `Cache/metaread.cpp` and `Cache/reader.cpp` top-of-file comments for more detail.

------------------------------------------------------------------------------------------

------------------------------------------------------------------------------------------
Icon Caching
------------------------------------------------------------------------------------------

    • Each icon is a 256 px longside QIcon, so we want to balance how many we load when
      weighing performance vs memory footprint.

    • Icons are stored in the DataModel at dm->(row, 0).data(Qt::DecorationRole). The
      loaded status can be determined two ways:
        1. dm->itemFromIndex(dmIdx)->icon().isNull()      // used up to 2023
        2. dmIdx.data(Qt::DecorationRole).isNull()        // switched to in 2023

    • The number of icons to cache is governed by dm->iconChunkSize. How it is set is
      decided per folder in DataModel::resolveIconChunkSize (called from endLoad) and
      depends on the G::useJitIconCache flag.

    Brute force (G::useJitIconCache == false, the default)

        • dm->iconChunkSize is set to the DataModel row count, so every icon is cached
          regardless of folder size. Simplest behaviour; preferred for typical folders.
          (The G::memoryOverrunFlag bail in DataModel::setIcon1 remains the crash backstop.)

    Just-in-time, JIT (G::useJitIconCache == true)

        • A folder is cached fully only when its projected thumbnail footprint fits a memory
          budget; otherwise dm->iconChunkSize is clamped to a bounded sliding window and the
          existing machinery (setIconRange, MetaRead::needToRead, clearIconsOutsideChunkRange)
          loads and evicts icons as the user moves. JIT has three layers:

        Layer 1 - predictive budget (resolveIconChunkSize / iconBudgetCount)
            At folder load the per-icon footprint is estimated (worst case 256x256x4 bytes)
            and compared to a budget = free memory after a safety reserve and the image
            cache's own remaining claim (G::imageCacheHeadroomMB, published by
            ImageCache::memChk), times G::jitIconCacheMemFraction. Fits -> full; else ->
            bounded window. iconChunkSize is then clamped to the row count and to a hard
            floor (iconChunkFloor): max(256, 3 x visibleIcons). The floor OVERRIDES the
            budget — if the image cache has claimed most of memory and the budget would fall
            below 3x the visible icons, the floor wins and the thumbnails take the memory
            they need; the image cache self-throttles because its own ceiling is derived
            from G::availableMemoryMB, which drops as the thumbnails are held. The floor is
            re-applied by MW::updateIconRange whenever the visible count changes (e.g.
            switching to the denser Grid view). Note: sizing is purely budget + floor; there
            is no separate "visible x 3" target beyond this floor.

        Layer 2 - measured refinement (refineIconChunkSize)
            As real icons load, DataModel::setIcon1 accumulates their true size. After ~200
            icons the budget is recomputed with the measured average and the window may grow,
            or be promoted back to full, when thumbnails turn out smaller than the worst-case
            estimate. Grow-only and one-shot per folder.

        Layer 3 - defensive pressure valve (applyIconCachePressure)
            A 1 s GUI-thread timer watches memory pressure (DataModel::memoryPressureLevel,
            derived from G::availableMemoryMB). On warn it halves the window; on critical it
            clamps to the visible page and evicts. It never grows. A 60 s cooldown plus a
            2 GB available-memory high-water mark (hysteresis) gate when the latch releases,
            so the cache cannot thrash between shrink and re-grow. Releasing does not re-grow;
            Layer 1/2 re-decide on the next folder load. Brute force is not touched by the
            valve.

    • Independent of the above, MW::updateIconRange still grows dm->iconChunkSize to at least
      the number of visible icons when more thumbnails are visible (in thumbView, gridView or
      tableView) than the current chunk, e.g. after an app or thumbnail resize.

    • When dm->iconChunkSize changes or there is a scroll event, the metadata cache read
      process is rerun to update the icons.

------------------------------------------------------------------------------------------
Load Responsiveness (keeping the GUI usable during a large/recursive load)
------------------------------------------------------------------------------------------

    The cost of a folder load on the GUI thread is dominated not by decoding but by the
    flood of QStandardItemModel signals each row change emits, every one of which runs the
    proxy (SortFilter) plus all three views' (thumbView, gridView, tableView) handlers
    synchronously. On a ~9000-image recursive tree this was ~30 s of GUI-thread time,
    starving keystrokes/mouse for seconds. Three measures cut it to near zero:

    1. Batch per-row writes (always on). DataModel::setIcon1 and setValDm wrap their setData
       calls in a QSignalBlocker and emit ONE dataChanged, mirroring addMetadataForItem.
       Four/two signals per icon -> one.

    2. Notify only visible rows (G::useVisibleOnlyIconEmit, default true). setIcon1/setValDm
       emit their one dataChanged only when the row is within [firstVisibleIcon,
       lastVisibleIcon] (DataModel::iconRowVisible). Off-screen icons are stored without
       notification and paint correctly when scrolled to. ~9000 emits -> ~visible-count.

    3. Batch the folder insert (G::useBatchedFolderInsert, DEFAULT FALSE). DataModel::addFolder
       can add a folder's rows with a single setRowCount (one rowsInserted, one view relayout)
       and fill the cells with signals blocked + one dataChanged for the range, instead of
       one rowsInserted + ~20 dataChanged per file (the enumeration freeze). This is OFF by
       default: that single wide dataChanged over the whole inserted block makes the
       sorting-enabled proxy/tableView re-sort the block descending, so folders load Z-A
       instead of A-Z. The per-row loop preserves A-Z and load latency is unchanged (the
       per-icon win comes from useVisibleOnlyIconEmit, item 2). Leave false unless the
       reordering is fixed first.

    Items 1-2 are independent of brute force vs JIT — they help any mode. Set the flags
    false to restore the prior per-signal behavior.

How to Test Icon Caching
------------------------------------------------------------------------------------------

    Flags (Main/global.cpp) — flip at runtime via the debugger or a temporary code change:

      G::useJitIconCache          false = brute force (cache all); true = JIT layers active.
      G::jitIconCacheMemFraction  share of the free-memory remainder budgeted for thumbnails
                                  (default 0.5).
      G::iconPressureTestLevel    forces the Layer 3 pressure level for deterministic testing
                                  (see values below).
      G::useVisibleOnlyIconEmit   true = setIcon1/setValDm notify views only for visible rows.
      G::useBatchedFolderInsert   true = addFolder inserts a folder's rows in one batch
                                  (default false; true reorders the proxy Z-A — see above).

    Diagnostics — Help > diagnostics. The DataModel report shows the live state:
      iconChunkSize, icon cache mode (JIT window / brute force), icon loaded (row ranges),
      icon avg footprint (measured vs worst-case), icon budget (icons),
      icon chunk floor (3x vis), imageCacheHeadroomMB, iconChunkRefined, memoryPressureLevel,
      iconCachePressureLatched.

    Status indicator — the metadata thread "circle" (MW::updateMetadataThreadRunStatus)
    is colour-coded on success so the active icon-cache mode is visible at a glance:
      red     metadata thread running
      green   idle, brute force (G::useJitIconCache == false)
      blue    idle, JIT caching active (G::useJitIconCache == true)
      orange  idle, forced pressure (G::iconPressureTestLevel >= 2)  [precedence: orange > blue]
      yellow  idle, load failed

    Brute force
      • Set G::useJitIconCache = false and load any folder. Confirm "icon loaded" spans
        every row and "icon cache mode" = brute force (full).

    Layer 1 - predictive budget
      • Set G::useJitIconCache = true. A folder small enough to fit the budget caches fully;
        a large folder (or a lower G::jitIconCacheMemFraction) shows "icon cache mode" =
        JIT window with iconChunkSize < row count.

    Layer 2 - measured refinement
      • In JIT window mode, scroll/load past 200 icons. Watch the refineIconChunkSize log
        line; "icon avg footprint" switches to "(measured, n=...)" and iconChunkSize may grow
        or be promoted to full (iconChunkRefined = true).

    Layer 3 - pressure valve (use the test override so you need not starve the machine)
      G::iconPressureTestLevel values:
        -1  use the real signal (availableMemoryMB)
         0  normal AND memory recovered      -> latch releases after the 60 s cooldown
         1  warn                             -> window halved, latched
         2  critical                         -> window clamped to the visible page, latched
         3  normal BUT memory not recovered  -> reaches the release branch, held by high-water

      Suggested sequence (G::useJitIconCache = true, a windowed folder loaded):
        1. = 2   critical: iconChunkSize drops to the visible page, icons evicted, latched.
        2. = 3   pressure "cleared" but not recovered: after 60 s, still latched (roomy=false).
        3. = 0   recovered: latch releases on the next 1 s tick (iconCachePressureLatched=false).
        4. = -1  restore the real signal.
      Confirm each transition in the DataModel diagnostics and the applyIconCachePressure
      log lines.

SCROLLING

    When the user scrolls a datamodel view (thumbView, gridView or tableView), the
    scrollbar change signal is received by thumbHasScrolled, gridHasScrolled or
    tableHasScrolled. The other views are scrolled to sync with the source view's
    midVisibleRow. The first, mid and last visible items are determined in
    MW::updateIconRange and the metaReadThread is called to update the metadata and icons
    in the range (chunk).

    However, when the program syncs the views this generates more scrollbar signals that
    would loop. This is prevented by the G::ignoreScrollSignal flag. The same flag is also
    employed in fileSelectionChange, where visible views are scrolled to centre on the
    current selection.

------------------------------------------------------------------------------------------
Painting Delays
------------------------------------------------------------------------------------------

    When the user changes modes in MW (i.e. from Grid to Loupe), an IconView instance
    (either thumbView or gridView) can change state from hidden to visible. Since hidden
    widgets cannot be accessed, we need to wait until the IconView becomes visible and
    fully repainted before attempting to scroll to the current index.

    Also, when an IconView resize event occurs — triggered by a change in the gridView
    cell size (thumbWidth) that requires justification, a change in the thumbDock
    splitter, or a resize of the application window — we also need to wait until
    scrollbar repainting is completed.

    The last paint event is identified by calculating the maximum of the scrollbar range
    and comparing it to the paint event, which updates the range each time. With larger
    datasets (i.e. 1500+ thumbs) it can take a number of paint events and hundreds of ms
    to complete.

    NOTE: An earlier IconView::waitUntilScrollReady helper has been removed. The wait-
    for-paint logic now lives inline in the scroll-handling paths in IconView and MW.

------------------------------------------------------------------------------------------
Scrollbar Change Flow
------------------------------------------------------------------------------------------

    • The ScrollBar signal valueChanged triggers MW::thumbHasScrolled,
      MW::gridHasScrolled or MW::tableHasScrolled.

    • MW::updateIconRange updates the first, mid and last visible icons from
      all views to find the greatest range of visible icons by calling
      IconView::updateVisible and TableView::updateVisible. dm->iconChunkSize
      is adjusted if it is less than the visible icon range.

    • Concurrent metadata loading:
        MW::update(midVisibleIcon) is called and it loads any missing icons and cleans up
        any orphaned icons (not in the iconChunkRange).

------------------------------------------------------------------------------------------
Resizing Icons or the Icon Viewport
------------------------------------------------------------------------------------------

    When the IconView viewport is resized or the icon size is changed, the result is a
    scroll event. Since we want to keep the view centred on its original state, the middle
    or centre icon is tracked as dm->scrollToIcon. G::resizingIcons is set for the resize
    and the subsequent scroll event does not reset dm->scrollToIcon. MW::thumbHasScrolled
    or MW::gridHasScrolled resets G::resizingIcons to false.

    This allows the three datamodel views to be synced using dm->scrollToIcon.

    Persisted icon size (between sessions):
        IconView::assignedIconWidth is the reference width that rejustify() uses to
        compute thumbs-per-row and then justify the cell width to exactly fill the row.
        The justified result lands in iconWidth, which can differ from assignedIconWidth.

        MW::writeSettings saves gridView->iconWidth (thumbWidthGrid). On restore,
        MW::createGridView loads iconWidth and must also set
        gridView->assignedIconWidth = gridView->iconWidth; otherwise rejustify() uses a
        stale/uninitialised reference and the persisted size is discarded, making the
        grid icon size inconsistent between sessions. (thumbView is handled separately:
        rejustify() skips "Thumbnails" while G::isInitializing and MW::showEvent applies
        the size via setThumbSize().)

FOCUS PREDICTOR

    Focus Predictor uses a machine learning model to predict where in the image the user
    will want to review the image focus — for example, the eye of the subject. When
    enabled, and the loupe view is zoomed, as the user advances each image is panned to
    the predicted focus review location.

------------------------------------------------------------------------------------------
Architecture
------------------------------------------------------------------------------------------

    Data
        All the training images and a CSV file that records the focus point, type and
        path:
            /Users/roryhill/Documents/Documents - Quark/FocusPointTrainer

    Python
        Code used to create the model:
            /Users/roryhill/Projects/FocusPointTrainer
        • config.py
        • model.py
        • train_focus_model.py              (creates the model)
        • validate_focus_model.py

    Model
        focus_point_model at /Users/roryhill/Projects/FocusPointTrainer in two versions:
        .pth and .onnx.

    FocusPointTrainer
        A C++ class that creates the "labeling" data from the training images in
            /Users/roryhill/Documents/Documents - Quark/FocusPointTrainer/focuspoint.csv.

    FocusPredictor
        A C++ class that uses the model to predict the focus point for the current image.

------------------------------------------------------------------------------------------
Process
------------------------------------------------------------------------------------------

    1. The training data is created using FocusPointTrainer to append the focus point,
       type and path for each training image to focuspoint.csv.

    2. In Code, train_focus_model is run (run Python in terminal). This creates
       focus_point_model.pth and focus_point_model.onnx. The .pth file is the native
       torch format and is converted to .onnx — the format that OpenCV uses. This enables
       C++ to use the model with the OpenCV library.

    3. Copy focus_point_model.onnx to /Users/roryhill/Projects/Winnow64/DetectionModels.

    4. Use FocusPredictor::predict to return the normalized image coordinates for the
       predicted focus location.

------------------------------------------------------------------------------------------
Implementation
------------------------------------------------------------------------------------------

    When focus prediction is enabled (Toggle Pan to Focus) and the loupe view is zoomed,
    the ImageView pans to the predicted focus point. See:
        • end of ImageView::loadImage
        • ImageView::predictPanToFocus              (get focus coord and pan)

    When this occurs, the location of the ImageView viewport within the scene is signalled
    to IconView, which tells IconViewDelegate to show a rectangle in the thumbView
    thumbnail matching the viewport in the scene. See:
        • ImageView::predictPanToFocus              emits the rectangle and image aspect
        • IconView::loupeRect                       receives rect/aspect, updates delegate
        • IconViewDelegate::setVpRect

HELP SYSTEM

    All in-app help pages are HTML files loaded by HtmlWindow (Utilities/htmlwindow.cpp).
    Every page shares a single stylesheet, Docs/help.css, which is applied automatically
    by HtmlWindow via QTextDocument::setDefaultStyleSheet. To restyle every help page at
    once, edit help.css; no other files need to change.

------------------------------------------------------------------------------------------
Architecture
------------------------------------------------------------------------------------------

    Utilities/htmlwindow.cpp        HtmlWindow widget (a QScrollArea hosting a
                                    QTextBrowser). Loads the HTML and applies
                                    Docs/help.css as the document's default stylesheet,
                                    and sets a qrc base URL on the document so relative
                                    <img src="images/..."> resolves.

    Docs/help.css                   Shared dark-theme stylesheet for every help page.
                                    Defines body, h1-h3, a, code, pre, .highlight, .warn.

    Docs/_template.html             Reference template demonstrating every supported
                                    element (headings, paragraphs, span colors, link,
                                    image, lists, code block, table). Copy this when
                                    creating a new page.

    Docs/<topic>.html               One file per help topic. The <head> only needs:
                                        <link rel="stylesheet" href="help.css">
                                    Qt ignores the <link> (it already loaded help.css
                                    as the default stylesheet); VS Code's Live Preview
                                    honors it, so the editor view matches Winnow.

    Docs/images/                    Image assets used by help pages. Reference with a
                                    relative path: <img src="images/Foo.png">.

    winnow.qrc                      Every .html and image referenced from help must be
                                    listed here so it is bundled into the binary.

    HtmlWindow lifecycle            Constructor sets Qt::Window (top-level with title bar)
                                    and Qt::WA_DeleteOnClose (self-cleanup on user close).
                                    Callers pass a parent (`window()` for embedded
                                    widgets, `this` for top-level dialogs) so the help
                                    window is destroyed with its owner. MW::closeEvent
                                    also sweeps findChildren<HtmlWindow*>() as a safety
                                    net.

------------------------------------------------------------------------------------------
Adding a New Help Page
------------------------------------------------------------------------------------------

    1. Copy Docs/_template.html to Docs/<yourpage>.html.

    2. Edit in VS Code (see "VS Code Workflow" below). Keep the existing
       <link rel="stylesheet" href="help.css"> in the <head>; only add a <style> block
       if you need per-page overrides.

       OR use BBedit.

    3. Place any images in Docs/images/ and reference them with a relative path,
       e.g. <img src="images/MyDiagram.png">.

    4. Register the new files in winnow.qrc under the existing /Docs prefix:
           <file>Docs/<yourpage>.html</file>
           <file>Docs/images/<YourImage>.png</file>     (if any)

    5. Open it from C++. Pattern (mirrors Filters::howThisWorks at filters.cpp:1767):

           #include "Utilities/htmlwindow.h"

           void MyWidget::someHelp()
           {
               QRect r = QRect(mapToGlobal(geometry().topLeft()), geometry().size());
               new HtmlWindow("Winnow - My Topic",
                              ":/Docs/yourpage.html",
                              QSize(700, 600), r, window());
           }

       The fifth argument (`window()` or `this`) sets the parent so the help window
       closes with its owner. Do not store the returned pointer or delete it manually;
       WA_DeleteOnClose handles cleanup.

    6. Build. No other wiring is required.

------------------------------------------------------------------------------------------
Editing Help Formatting Globally
------------------------------------------------------------------------------------------

    Edit Docs/help.css. Every help page picks up the change on the next launch (the .css
    is loaded fresh by HtmlWindow each time a help page opens).

    Constraint: QTextBrowser supports only Qt's HTML 4 / CSS 2.1 subset
        https://doc.qt.io/qt-6/richtext-html-subset.html
    Avoid flexbox, grid, CSS custom properties, and HTML5 semantic tags — Qt silently
    ignores them.

    For per-page overrides, add a <style> block to that page's <head>. It cascades on
    top of help.css.

------------------------------------------------------------------------------------------
VS Code Workflow
------------------------------------------------------------------------------------------

    Recommended editor: VS Code (free) with the Microsoft "Live Preview" extension.
    Reasons:
      • Plain-text HTML editing — will not silently inject CSS3 that Qt ignores
        (which is why WYSIWYG editors like Typora and Pinegrow are NOT recommended).
      • Side-by-side browser preview that refreshes on save.
      • The <link rel="stylesheet" href="help.css"> in each page makes the preview
        match Winnow's rendering closely.

    Mac-native alternative: BBEdit (free tier) — same plain-text philosophy.

    Expected differences between VS Code preview and Winnow:
      • Font sizes will not match pixel-for-pixel (Chromium vs QTextBrowser defaults).
        help.css sets explicit sizes on body/h1/h2/h3 to narrow the gap.
      • Anything outside the CSS 2.1 subset (above) will render in VS Code but not in
        Winnow. When in doubt, check in Winnow.
