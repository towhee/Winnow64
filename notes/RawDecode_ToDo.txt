RAW DECODE / DEVELOP - TASK LIST
================================================================================
Goal: decode, render and edit raw files natively in Winnow (no exiftool/libraw at
runtime), each format validated byte-identical to libraw. All gated by G::useRaw.

PERSISTED 2026-06-24. The original list lived in the "Raw decoder and editor"
conversation and was never saved to a file; this restores it from the project
record + current code. Statuses verified against the tree on this date.

Pipeline (ImageFormats/Raw/rawformat.cpp Decode):
  UnpackCfa (per-format) -> SubtractBlack -> Demosaic -> RawColor -> Develop -> OutputTransform

15   [done]      Implement the real Develop ops on the scene-linear buffer.
                 DONE (2026-06-25): all ten Basic sliders live (Develop/develop.{h,cpp}).
                 POINT ops (WB temp/tint, exposure, contrast, tone regions
                 highlights/shadows/whites/blacks) = a single FUSED, QtConcurrent-parallel
                 per-pixel pass: WB+exposure fold into a per-channel linear gain; contrast +
                 tone regions bake into a 1-D perceptual (~gamma-2.2) LUT (PointCoeffs::toneLut)
                 so the hot loop is one encode pow + one lookup. SPATIAL ops own a full pass
                 after the point pass: Texture (#6, mid-freq luma local contrast) and Dehaze
                 (#7). Denoise stays spatial/global-only. Radii scale with the image edge so
                 the proxy preview matches full-res. See Documentation.txt "DEVELOP / IMAGE
                 EDIT" + "BUILD PHASING".
                 CONSIDER (dehaze quality): Dehaze is currently an APPROXIMATION
                 (large-radius local contrast + low-pivot contrast + saturation; user-chosen
                 2026-06-25). Consider a true DARK-CHANNEL-PRIOR dehaze as a follow-up:
                 estimate atmospheric light + per-pixel transmission from the dark channel,
                 refine the transmission map with a guided filter, then recover scene
                 radiance. ~120-150 lines, heavier on the full-res settle render, more
                 parameters to tune; physically removes real haze (better on hazy
                 landscapes). Would slot into Develop::Dehaze behind the same slider.

19   [wip]       Harden per-file WB / black / white / CFA reads for the DECODED
                 vendors. Audited 2026-06-24 against the libraw oracle over the
                 _ThumbTest corpus (iCloud /_ThumbTest).
                 ROBUST already (read tags, match oracle): Nikon, Sony, Panasonic, DNG.
                 DONE this pass:
                   - Olympus CFA: now read from EXIF CFAPattern (0xA302) instead of a
                     hardcoded BGGR. Validated: E-M1=BGGR, OM-1=RGGB (the OM-1 was
                     rendering with red/blue swapped).
                   - Fuji black/white: now read from the FujiIFD that prefixes the CFA
                     data block (cfaOff -> little-endian TIFF -> 0xf000 -> FujiIFD;
                     BitsPerSample 0xf003, per-CFA BlackLevel 0xf00a). Replaces the
                     hardcoded 1024/16383. Verified at parse level on GFX(64)/X-T2(1022)/
                     X-T50(1023). The compressed RAF decoder (#11) now also reads 0xf007
                     StripOffset to locate the compressed bitstream, so these levels take
                     effect on the compressed X-T2/GFX/X-T50 corpus.
                   - Canon white: was (1<<precision)-1 = 16383 but the 7D Mark II
                     saturates at 13584 -> washed highlights. cameramatrix.cpp now also
                     carries libraw adobe_coeff t_black/t_maximum per model
                     (cameraLevelsForModel); Canon uses t_maximum when set. Verified
                     7D2 -> 13584. The generator (tools/gen_cameramatrix.py) emits the
                     levels too; regen pulls future bodies.
                 REMAINING (minor):
                   - Olympus black 255 vs OM-1's 254; Sony default black 512 vs an 800
                     sample; verify Canon ColorData WB offset across versions. Other
                     vendors could also adopt cameraLevelsForModel where they hardcode.
